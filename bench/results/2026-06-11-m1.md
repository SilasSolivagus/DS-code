## 解读（人工分析）

**成功率打平（名义 7/10 vs 7/10），但 s3 是场景设计缺陷**：s3 的 prompt 没要求"引用文件"，评分却要求命中 `bash.ts`——deepcode 两次都正确讲清了 printf/$PWD marker 机制（命中机制关键词）只是没报文件名；CC 第 1 次连机制都没讲对。剔除文件名要求后实际为 deepcode 9/10、CC 8/10。s3 的 prompt 已修正（加"引用具体文件"），下轮生效。

**token 经济性是数量级差距**：五场景中位输入合计 deepcode ~45k vs CC ~773k（**约 17 倍**）；未命中缓存的"真金"输入 deepcode 每场景 2-4k vs CC 17-53k。CC 的 37k 固定提示词开销 × 多轮重发是主因。

**速度**：deepcode 中位 9-18s，CC 131-290s。注意 CC 墙钟含异常高的 CLI 启动开销（本机实测纯 API 时间仅占小头，疑与代理环境有关），不能全部归因于 harness 轮次，但量级差距是真实的。

**质量细节**：CC 在 s2 有一次没找到 matchRule（Claude 调教的提示词在 DeepSeek 上偶发探索不足）；deepcode 在 s1 有一次讲对了内容但没引用 api.ts（引用纪律待加强——记入待调 prompt 清单）。

**结论（M1 阶段，只读问答场景）**：答案质量两边相当，deepcode 以 ~1/17 的 token 和 ~1/10 的耗时达到同等效果。CC 的 harness 优势（subagent/compact/丰富工具）在这类短任务中用不上，纯付固定开销。真正的考验在 M2 写任务和 M4 长任务——届时 CC 的成熟度可能反超，继续对照。
