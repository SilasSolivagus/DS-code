# 1.5 可写子代理 + git worktree 隔离 设计

日期：2026-06-25
对齐目标：Claude Code 的「可写子代理 + git worktree 隔离」机制（EnterWorktree/ExitWorktree 工具 + Agent `isolation:"worktree"` 字段 + settings.worktree + 非 git 仓库 hook 兜底）。**经两轮 opus 专家逐字实读 CC bundle（`@anthropic-ai/.claude-code-2DTsDk1V/cli.js`，v2.1.76）核对**，证据片段带 byte 偏移可复核。
所属：master roadmap 第 1 层 1.5（L→XL，独立批）。执行序 #2。
用户拍板：**完全照搬 CC**（除两处 deepcode 架构无对应物：无 Bash sandbox、子代理一直非交互——见 §4 映射）。

## 1. 背景与目标

deepcode 现状：**子代理强制只读**——`GLOBAL_SUBAGENT_DENY = ['Edit','Write','Agent','NotebookEdit','ExitPlanMode']`（agentTypes.ts:19）把所有写工具与递归剥掉；子代理 `ToolContext.setCwd` 冻结为 no-op（subagentRunner.ts:84「子代理只读，不许漂移主 cwd」）；子代理无审批 UI，`subagentPermissionDecision`（agent.ts:18）安全命令自动放行、危险命令拒绝。

本批 = **让子代理可写，并以 git worktree 作为隔离/安全机制**，对齐 CC 三个独立表面：
1. **会话级 EnterWorktree / ExitWorktree 工具**——主 agent 自己切进隔离 worktree 工作。
2. **子代理 `isolation:"worktree"` 字段**——派子代理时建独立 worktree，子代理可写、cwd 锚定 worktree，结束后无改动自动删、有改动保留并回传 path+branch。
3. **settings.worktree 配置**（symlinkDirectories / sparsePaths）+ 非 git 仓库的 WorktreeCreate/WorktreeRemove hook 兜底。

产品价值：可写子代理 + worktree 隔离是 deepcode 团队/云愿景（[[deepcode-vision]]）的能力底座——多个可写 agent 在隔离副本里并行干活、再 review/merge 回主树。

## 2. CC 真实机制（实读核实，含证据锚点）

### 2.1 会话级 EnterWorktree / ExitWorktree（读①）
- **EnterWorktree**（`sP1`）：input `{name?: string}`（缺省随机名）。`isReadOnly()=false`、`checkPermissions→allow`（无弹窗）。`call`：在 git 仓库内 `git worktree add -b worktree-<name> <gitroot>/.claude/worktrees/<name> <base>`，base=`origin/<默认分支>`（解析不到回退 HEAD）。创建后 `process.chdir` + `setIsInWorktree(true)` **切换会话 cwd**。返回 `{worktreePath, worktreeBranch?, message}`。已在 worktree 会话再调 → 抛 `Already in a worktree session`。
- **ExitWorktree**（`tP1`）：input `{action: "keep"|"remove", discard_changes?: boolean}`。改动检测 `n8q`：`git -C <wt> status --porcelain` 统计未提交文件数 + `git -C <wt> rev-list --count <originalHead>..HEAD` 统计领先提交数。`validateInput`：`action:"remove"` 且 `!discard_changes` 且有改动 → **拒绝**（列出文件/提交/分支）。无改动 remove → `git worktree remove --force` + `git branch -D`。keep → 保留磁盘，恢复 cwd。无活跃会话 = no-op。

### 2.2 子代理 isolation:"worktree"（读①）
- Task/Agent input：`isolation: C.enum(["worktree"]).optional()`，description「creates a temporary git worktree so the agent works on an isolated copy of the repo」。**与 `cwd` 字段互斥**。
- 运行时：`if(isolation==="worktree") s=await zl6("agent-"+id.slice(0,8))` 建独立 worktree（`.claude/worktrees/agent-<id8>`，分支 `worktree-agent-<id8>`）；子代理整段执行包进 `dHA(worktreePath, ...)`（AsyncLocalStorage 设 cwd=worktree）；注入 worktree 系统提示 `H_4`。
- 收尾：无改动（`pu8`）→ `E66` 删 worktree+branch，结果不带 path；有改动 → 保留，结果回传 `{worktreePath, worktreeBranch}`。

### 2.3 子代理权限/可写模型（读②，定论）
- 子代理 `toolPermissionContext` = **浅拷父代理**，只覆盖 `mode`，默认 `acceptEdits`（来自 agent frontmatter `permissionMode`，缺省兜底）。
- 审批分叉：**同步前台**子代理 `canShowPermissionPrompts` 默认 true → ask **能弹 UI**（复用父 canUseTool）；**异步/后台**子代理 `shouldAvoidPermissionPrompts:true` → auto-classifier(`uYz`) → 不放行即 **deny**（reason `asyncAgent:"Permission prompts are not available in this context"`，证据 byte 10575635）+ 连续 deny 熔断 abort。
- acceptEdits 精确语义：只无脑放行 ①目录内文件写（`kI(path,ctx)`，路径在外则 ask）②7 个白名单 base 命令 `["mkdir","touch","rm","rmdir","mv","cp","sed"]`（证据 byte 10544643，base command 级，不看参数）；其余落正常权限 + Bash 安全检查器 + 危险动作分类器。
- 子代理工具池 `u66/FX`：含 **Edit/Write/Bash + Agent/Task 本身**（**CC 允许递归派子代理**，仅拦 fork-in-fork）；硬剔除全局名单极小 = MCP 资源工具(`Ll`/`hl`) + StructuredOutput(`oM`)，**不剔 Edit/Write/Bash**（证据 byte 8979126）。
- 危险命令钳制：Bash 安全检查器集合（misparsing/注入，证据 byte 5457700）+ LLM `dangerous-agent-action` 分类器，**非静态黑名单**。
- worktree 子代理权限**不变**，仅 cwd 经 AsyncLocalStorage 切到 worktree，写围栏靠「cwd 即 worktree」天然生效，**无显式 additionalWorkingDirectories 注入**。

### 2.4 settings.worktree + 非 git hook（读①）
- `worktree: { symlinkDirectories?: string[], sparsePaths?: string[] }`。创建后 `uu8` 应用：拷 `settings.local.json`、配 `core.hooksPath` 指主仓 hooks、`symlinkDirectories` 建符号链接、`sparsePaths` 走 `git worktree add --no-checkout` + `git sparse-checkout set --cone -- <paths>` + `git checkout HEAD`。还读 `.worktreeinclude` 拷 gitignored 目录（schema 未列的额外机制）。
- 非 git 仓库：`iN1()` 检测配了 WorktreeCreate hook → `nN1(name)` 触发，hook stdout.trim() 作 worktreePath；WorktreeRemove `rN1(path)` payload `{hook_event_name, worktree_path}`。两者都没配且非 git → 抛错。

## 4. deepcode 忠实映射（两处架构无对应物）

CC 有、deepcode 无对应物的两点，按「最贴近 CC 语义」映射，非自作主张偏离：

- **子代理非交互 = CC async 子代理路径**：deepcode 子代理一直无 canUseTool 冒泡 UI（架构既定）。CC 的 async 子代理同样 `shouldAvoidPermissionPrompts` → auto-classifier → 不放行即 deny。映射：deepcode 现有 `subagentPermissionDecision`（isDangerous→no / 否则→yes）**就是** deepcode 版 auto-classifier，本批沿用不改。「加前台子代理→主 UI 冒泡」是另一个巨型件，**明确不在本批**。
- **Bash 无 sandbox**：CC 可写子代理的部分安全来自 Bash sandbox（seatbelt）。deepcode Bash 用 `execFile('/bin/bash')` 无沙箱（bash.ts:90）。worktree 只让**文件工具**（Edit/Write/Read/Glob/Grep）cwd 指向隔离副本；**Bash 用绝对路径/`cd ..` 仍可写主树**。D4 决定不补围栏，**spec 如实记录此 gap**，不夸大 worktree 对 Bash 的隔离强度。
- **acceptEdits 白名单差异**：CC acceptEdits 对 Bash 只放行 7 个 base 命令。deepcode 子代理 Bash 走 `subagentPermissionDecision`（isDangerous→no/否则→yes），比 CC acceptEdits **更宽**（非危险命令全放行）。这是 deepcode 既有行为，本批不收紧（改它=独立决定，超范围）。

## 5. 组件设计

### 5.1 新模块 `src/worktree.ts`（git 操作底座）

deepcode 现无 git 模块。新建纯 `execFile('git', …)` 封装（async，不引第三方 git 库——对齐项目「先搜内置」原则，git CLI 即够）。命名落 deepcode 命名空间：worktree 根 `<gitroot>/.deepcode/worktrees/<name>`（CC 用 `.claude/`），分支 `worktree-<name>` / 子代理 `worktree-agent-<id8>`。

```ts
export interface WorktreeHandle { worktreePath: string; worktreeBranch: string; headCommit: string; gitRoot: string }

/** git 仓库主工作树根（解析 linked worktree 也回主仓 commondir 的工作树）；非 git 返回 null。 */
export async function resolveGitRoot(cwd: string): Promise<string | null>

/** 建 worktree：git worktree add -b worktree-<name> <root>/.deepcode/worktrees/<name> <base>。
 *  base = 解析 origin/<默认分支>（git symbolic-ref refs/remotes/origin/HEAD），回退 HEAD。
 *  applyConfig=true 时应用 settings.worktree（symlink/sparse）+ 拷 settings.local。 */
export async function createWorktree(gitRoot: string, name: string, cfg?: WorktreeConfig): Promise<WorktreeHandle>

/** 改动检测：status --porcelain 非空 ‖ rev-list --count <headCommit>..HEAD > 0。 */
export async function worktreeChanges(worktreePath: string, headCommit: string): Promise<{ changedFiles: number; commits: number }>

/** 删 worktree：git worktree remove --force <path> + git branch -D <branch>（吞「已删」错误）。 */
export async function removeWorktree(h: Pick<WorktreeHandle, 'worktreePath' | 'worktreeBranch' | 'gitRoot'>): Promise<void>

export interface WorktreeConfig { symlinkDirectories?: string[]; sparsePaths?: string[] }
```

- **gitRoot 锚定**：worktree 目录恒挂主仓工作树根（用 `git rev-parse --git-common-dir` 反推主工作树），即便从 linked worktree 内创建嵌套 worktree（递归子代理场景）也不会嵌进 `<worktree>/.deepcode/worktrees`。
- **base 解析**：`git symbolic-ref --short refs/remotes/origin/HEAD` → 失败用 `git rev-parse --abbrev-ref HEAD` → 失败 `HEAD`。不做 CC 的 PR fetch 路径（YAGNI，本批无 PR 模式）。
- **sparse**：有 sparsePaths 时 `add --no-checkout` + `sparse-checkout set --cone` + `checkout HEAD`，失败回滚 `worktree remove --force`。
- **symlink**：`applyWorktreeConfig` 对每个 symlinkDirectories 项建 `worktree/<dir> -> mainRepo/<dir>` 符号链接（目标存在才建），并拷 `settings.local.json`。
- 全函数 fail-safe：git 命令非 0 退出抛带 stderr 的 Error，由调用方决定降级。

### 5.2 会话级 EnterWorktree / ExitWorktree 工具

新建 `src/tools/enterWorktree.ts`、`src/tools/exitWorktree.ts`，注册进 `allTools`。

**会话 worktree 状态**：新增「活跃 worktree 会话」持有器。挂 `ToolContext` 新可选字段（主会话/headless 注入，子代理不注入）：
```ts
// tools/types.ts ToolContext 增：
worktreeSession?: {
  get(): { originalCwd: string; worktreePath: string; worktreeBranch: string; headCommit: string; gitRoot: string } | null
  enter(s: …): void   // 切 cwd 由工具调 ctx.setCwd 完成，这里只记元数据
  exit(): void
}
```
useChat.ts / headless.ts 用闭包变量实现（不落盘——worktree 会话不跨 resume；对齐 CC mid-session 语义）。

- **EnterWorktree**：`isReadOnly:false`，`needsPermission:()=>false`（照搬 CC checkPermissions→allow，不弹窗）。`call`：`worktreeSession.get()` 非空 → 抛「已在 worktree 会话」；`resolveGitRoot(ctx.cwd())` 为 null → 抛非 git 提示（或走 hook，见 5.5）；`createWorktree` → `ctx.setCwd(worktreePath)` + `worktreeSession.enter(...)` + 发 `WorktreeCreate` hook 事件。返回 CC 同款 message。
  - **plan 模式**：EnterWorktree 非只读 → plan 模式被 checkPermission 的 plan 门拒（permissions.ts:232），符合预期（plan 只读，不该建 worktree）。
- **ExitWorktree**：`isReadOnly:false`，`needsPermission:()=>false`。`call`：`worktreeSession.get()` 为 null → no-op 提示；算 `worktreeChanges`；`action:"remove"` 且有改动且 `!discard_changes` → 返回 CC 同款拒绝文案（**不抛错，返回引导文本**——deepcode 工具约定返回字符串）；否则 remove 走 `removeWorktree` + 发 `WorktreeRemove` hook；keep 仅恢复 cwd。最后 `ctx.setCwd(originalCwd)` + `worktreeSession.exit()`。

### 5.3 子代理 isolation:"worktree"（核心 cwd 重构）

**Agent 工具 schema**（agent.ts）增：
```ts
isolation: z.enum(['worktree']).optional().describe('隔离模式。"worktree" 在临时 git worktree 里跑子代理，给它仓库的隔离副本。无改动自动清理；有改动则返回 worktree 路径与分支。'),
```

**ToolContext cwd 重构**（roadmap 点名）：子代理子 ctx 从「共享父 cwd getter + 冻结 setCwd」改为**独立 cwd 状态**：
```ts
// subagentRunner.ts runSubagent：worktree 模式下子 ctx 独立 cwd
let subCwd = opts.worktreePath ?? ctx.cwd()      // 普通子代理 = 父 cwd 快照；worktree 子代理 = worktree 路径
const subCtx: ToolContext = {
  cwd: () => subCwd,
  setCwd: d => { subCwd = d },   // 子代理内 Bash `cd` 现可在自身 cwd 漂移（不再污染主 cwd——本就是独立变量）
  …
}
```
- 普通子代理（无 isolation）：`subCwd = ctx.cwd()` 快照，`setCwd` 可漂移**自身** subCwd（不再 no-op，但因独立变量也不污染主 cwd）。这修正了「子代理 Bash cd 被静默吞」的现状，且语义更对（子代理有自己的工作目录）。
- worktree 子代理：`subCwd = worktreePath`。文件工具 `path.resolve(cwd, file)` 天然落 worktree（对齐 CC「cwd 即 worktree」）。

**Agent.call 接线**（前台 + 后台两路）：
```ts
const wantWorktree = input.isolation === 'worktree'
let wt: WorktreeHandle | null = null
if (wantWorktree) {
  const root = await resolveGitRoot(ctx.cwd())
  if (!root) { /* 非 git：走 hook 兜底或抛错，见 5.5 */ }
  wt = await createWorktree(root, `agent-${id.slice(0,8)}`, settings.worktree)
}
// runSubagent 传 worktreePath=wt?.worktreePath；systemPrompt 末尾追加 worktree 提示（5.4）
// 收尾：
if (wt) {
  const ch = await worktreeChanges(wt.worktreePath, wt.headCommit)
  if (ch.changedFiles === 0 && ch.commits === 0) { await removeWorktree(wt) }
  else { /* 保留；把 worktreePath+branch 附加进返回给主代理的文本 */ }
}
```
- 后台路径在脱钩 async 内同样建/清 worktree（finally 兜底清理，abort 时按有无改动决定删/留）。
- **`isolation` 与 deepcode 无 `cwd` 字段**：deepcode Agent 工具暂无 `cwd` input 字段（CC 有且与 isolation 互斥）。本批不加 `cwd` 字段（YAGNI），故无互斥冲突。

### 5.4 可写子代理（GLOBAL_SUBAGENT_DENY 翻转 + 系统提示）

- `GLOBAL_SUBAGENT_DENY` 从 `['Edit','Write','Agent','NotebookEdit','ExitPlanMode']` → **`['ExitPlanMode']`**（移除 Edit/Write/NotebookEdit 使可写；移除 Agent 放开递归——见 5.6；保留 ExitPlanMode：子代理无 plan 模式 UI，照搬 CC 不让子代理 ExitPlanMode）。
- 内建 agent 类型保持读写边界：`Explore`/`Plan` 的 `disallowedTools` 已含 `['Edit','Write','Agent']` → 仍只读（类型级 deny，对齐 CC 靠 def 限制）。**注意**：Explore/Plan 的 disallowedTools 也要相应处理 Agent——它们本就该只读不递归，保留。`general-purpose`（`tools:['*']`）现可写 + 可递归。
- **worktree 系统提示**：worktree 子代理在 `def.getSystemPrompt()` 后追加 deepcode 版隔离提示（对齐 CC `H_4` 语义，中文化）：
  > 你在一个隔离的 git worktree 里工作：`<worktreePath>`——同一仓库、同样的相对文件结构、独立工作副本。继承上下文里的路径指向父代理的工作目录，需翻译到你的 worktree 根。编辑前先重读文件（父代理可能已改动）。你的改动只留在此 worktree，不影响父代理的文件。

### 5.5 settings.worktree 配置 + 非 git hook 兜底

- **config schema**（config.ts / settingsLayers.ts）增 `worktree?: { symlinkDirectories?: string[]; sparsePaths?: string[] }`。**不进 DANGEROUS_TOP_KEYS**（D3：不剥离，全层生效）。`parsePresent` 注册新字段（3.9 约定：白名单新字段才被保留）。
- **非 git hook 兜底**：`resolveGitRoot` 为 null 时，若配了 `WorktreeCreate` hook → 发 hook，stdout.trim() 作 worktreePath（headCommit 置空 → 收尾不做 git 改动检测，按 hook-based 保留）；ExitWorktree/子代理清理对 hook-based worktree 发 `WorktreeRemove`。都没配 → 抛 CC 同款错误文案。
- **hook 事件**：`WorktreeCreate { hook_event_name, name }`、`WorktreeRemove { hook_event_name, worktree_path }` 加进 hooks.ts 的 HookEvent 联合 + hookRuntime 透传。

### 5.6 递归子代理防死锁（D5 必要配套）

放开 Agent 递归后，deepcode `MAX_ACTIVE=4` 阻塞信号量会死锁（4 槽全是父代理 acquire 后等子代理，子代理 acquire 永远抢不到槽）。重构 subagentRunner 信号量：

- `ToolContext` 增 `subagentDepth?: number`（主会话/headless 注入 0；子 ctx 注入 `depth+1`）。
- **顶层子代理**（depth 0 → 派出的子代理 depth 1）：维持 acquire/release 占 4 槽主池（现行为不变）。
- **递归子代理**（父 depth ≥ 1）：**不占主池信号量**——父已占一槽，子不再竞争同池（消除死锁根因）；改受 **`MAX_SUBAGENT_DEPTH`（建议 3）** 上限约束：`depth >= MAX_SUBAGENT_DEPTH` 时 Agent.call 直接抛 `已达子代理递归深度上限（<N>），请用现有工具直接完成`（对齐 CC fork-in-fork 拦截的「直接干」语气）。
- **运行中子代理总数硬上限**（runaway backstop，非阻塞）：全局计数 `MAX_TOTAL_ACTIVE`（建议 32），超则 Agent.call 抛错拒绝新子代理（防扇出爆炸）。
- 记忆 fork 信号量（`acquireMemory`，MAX_MEMORY_ACTIVE=2）不动。

> 此重构是「放开递归」的安全前提，单独成任务（架构件，需 opus 终审）。

## 7. 真机冒烟（碰会话 cwd 切换 + TUI 状态栏，必做）

deepcode 默认全屏跑 FullscreenApp。冒烟项（家目录建真实 git 小仓库，**别放 harness scratchpad**——沙箱私有用户终端读不到）：
1. EnterWorktree → 状态栏/页脚 cwd 切到 worktree、`git branch` 显示 `worktree-xxx`、建文件。
2. ExitWorktree keep → cwd 恢复、worktree 保留有改动。
3. ExitWorktree remove（无改动）→ 删干净；remove（有改动且无 discard）→ 拒绝文案；discard_changes → 删。
4. Agent isolation:"worktree" 子代理写文件 → 主树不变、worktree 有改动并回传 path+branch；无改动子代理 → worktree 自动删。
5. 递归：general-purpose 子代理再派子代理（深度 2）正常完成、深度超限报错、不死锁。
6. settings.worktree.symlinkDirectories=['node_modules'] → worktree 内 node_modules 是符号链接。
