# 5.6 权限弹窗 deny / 来源层级展示（PermissionDecisionReason）设计

日期：2026-06-22
对齐目标：Claude Code `PermissionRuleExplanation` / `PermissionDecisionDebugInfo` / `PermissionDecisionReason`。
所属：master roadmap 第 5 层体验件，C3 收尾批 TUI 破例两件之一。

## 1. 背景与目标

当一个工具调用被权限系统拦下时，用户应当看到**为什么**、**命中了哪条规则**、**规则来自哪个来源层级**。CC 通过 `PermissionDecisionReason`（可辨识联合）把决策上下文携带到弹窗与拒绝消息中，渲染成「`Bash(npm install)` rule from user settings …」。

deepcode 现状（亲读 `src/permissions.ts` / `src/deny.ts` / `src/settingsLayers.ts` / `src/tui/components/PermissionDialog.tsx`）：

- `checkPermission` 返回 `{ ok: true } | { ok: false; reason: string }`——deny 有 reason 文本（`路径被 deny 规则拒绝（${hit}）`），但**不带来源**。
- `PermissionContext.ask(toolName, desc)`——弹窗拿不到命中规则。
- `matchRule` / `bashCommandAllowed` 只返 boolean，不返命中的规则。
- `settingsLayers` 的 provenance **只追踪顶层 key**（`permissions`），**不追踪数组元素**，无法知道某条 `allow[i]`/`deny[j]` 来自哪个 scope。
- `PendingAsk { toolName, desc, dangerous, resolve }`——无 reason/source。
- `PermissionDialog` 渲染标题 / 高危警告 / diff / 三态菜单，不展示规则与来源。

**目标**（用户钦定，全部"完整对齐 CC"）：

1. 引入 `PermissionDecisionReason` 可辨识联合（镜像 CC，剪到 deepcode 实有变体）。
2. settingsLayers 做 **per-rule provenance**（allow + deny 都追踪，完整底座）。
3. 来源粒度做到**完整 scope 级**：`builtin | user | project | local | flag`。
4. deny 来源展示**两面覆盖**：① Bash deny → 降级 ask 的**弹窗**；② Read/Edit/Write deny → **硬拒绝文本**。
5. checkPermission 的 **allow 路径也 attach decisionReason**（完整镜像 CC；现阶段无消费方，预铺将来 `/permissions` UI）。

## 2. 范围裁剪（N/A，不建死分支）

deepcode 无对应子系统，故 CC 这些 `PermissionDecisionReason` 变体不纳入：
`mode`（deepcode 默认即 ask，yolo/acceptEdits 静默放行无展示面）、`classifier`、`subcommandResults`、`sandboxOverride`、`safetyCheck`、`workingDir`、`asyncAgent`、`permissionPromptTool`。
CC 的 source 值 `policySettings`/`cliArg`/`command`/`session` 同样 N/A（deepcode 无这些层；`flag` 对应 `--settings` 标志层）。

复合 Bash 命令（`bashCommandAllowed` 拆段授权）不引入 CC 的 `subcommandResults`：deny 命中点取**首个命中的 deny 段规则**，allow 命中取精确全量规则或首个命中段规则，收敛为单个 `rule`。

## 3. 数据结构（新增于 `src/permissions.ts`）

```ts
export type PermissionRuleSource = 'builtin' | 'user' | 'project' | 'local' | 'flag'

export interface PermissionRule {
  source: PermissionRuleSource
  behavior: 'allow' | 'deny'
  value: string   // 规则字符串，如 'Bash(npm test:*)' 或 deny pattern '~/.ssh/**'
}

export type PermissionDecisionReason =
  | { type: 'rule'; rule: PermissionRule }
  | { type: 'hook'; hookName: string; reason?: string }
  | { type: 'other'; reason: string }
```

## 4. 组件设计

### 4.1 settingsLayers — per-rule provenance

`mergeScopePartials` 在现有 `for (const { scope, partial } of layers)` 遍历里**额外**累积每条 allow/deny 规则的来源，产出：

```ts
permissionSources: {
  allow: Record<string, SettingScope>
  deny: Record<string, SettingScope>
}
```

- 规则字符串 → 贡献它的 scope。
- 同一规则出现在多个 scope → 归属**最高优先级 scope**（层序 user < project < local < flag，与 deepmerge "后者胜" 一致）。
- 因 `stripUntrustedScope` 剥离 project / git-tracked-local 的 `permissions.allow`：allow 源实际 ∈ {user, 非git-local, flag}；deny 保留 → 源 ∈ {user, project, local, flag}。
- `LayeredResult` 增 `permissionSources` 字段（纯新增，向后兼容）。
- 测试覆盖：单 scope、多 scope 重复规则归属、allow 被剥后只剩可信源、空配置。

### 4.2 deny.ts — builtin 源合并

新增：

```ts
export function buildDenySourceMap(
  configDenySources?: Record<string, PermissionRuleSource>,
): Record<string, PermissionRuleSource>
```

`BUILTIN_DENY` 每条 → `'builtin'`，并入 config deny 的 scope 源。重叠归属规则（明确）：先写 builtin，再写 config，**config 同名覆盖 builtin**（与 `resolveDenyList` 顺序及"用户配置更具体"一致）。`resolveDenyList` 保持不变。

### 4.3 permissions.ts — 命中规则查找 + 决策原因

- 新增 `findMatchingRule(rules, toolName, desc): string | null`、`findBashMatchingRule(command, rules): string | null`：复用现有 `matchRule`/`bashCommandAllowed`/`matchExactRule`/`splitBashCommand` 逻辑，返回命中的规则字符串而非 boolean。复合命令：先试精确全量规则，否则返回首个命中段的规则。`matchRule`/`bashCommandAllowed` 保留（内部可改为基于 find 派生 boolean，零行为变化）。
- `PermissionContext` 增：
  ```ts
  ruleSources?: Record<string, PermissionRuleSource>
  denySources?: Record<string, PermissionRuleSource>
  ```
- `ask` 签名扩为 `ask(toolName, desc, reason?: PermissionDecisionReason) => Promise<Decision>`。
- `checkPermission` 返回类型扩为：
  ```ts
  Promise<
    | { ok: true; decisionReason?: PermissionDecisionReason }
    | { ok: false; reason: string; decisionReason?: PermissionDecisionReason }
  >
  ```
- deny 路径：
  - 捕获命中 pattern `hit`，源 = `pc.denySources?.[hit] ?? 'builtin'`。
  - 非 Bash 硬拒绝：reason 文本升级为 `路径被 deny 规则拒绝（${hit}，来自 ${permissionSourceName(src)}）`，decisionReason = `{type:'rule', rule:{source, behavior:'deny', value:hit}}`，onDenied 也带新文本。
  - Bash forceAsk：把 `{type:'rule', rule:{source, behavior:'deny', value:hit}}` 透传给 `pc.ask`。
- allow 路径：经 findMatchingRule 拿命中规则，源 = `pc.ruleSources?.[rule] ?? 'user'`，返回 `{ ok:true, decisionReason:{type:'rule', rule:{source, behavior:'allow', value:rule}} }`。
- hook 拒绝：decisionReason = `{type:'hook', hookName:'PermissionRequest', reason}`。
- 用户拒绝：decisionReason = `{type:'other', reason:'用户拒绝了此操作'}`（可选）。

### 4.5 useChat.ts 接线

- 从 `loadLayeredSettings()` 取 `permissionSources`；`ruleSources = permissionSources.allow`；`denySources = buildDenySourceMap(permissionSources.deny)`；注入 `PermissionContext`。
- `PendingAsk` 增 `reason?: PermissionDecisionReason`。
- `pc.ask` 实现把第三参 `reason` 存进构造的 `PendingAsk`。
- "总是允许" saveRule 写 user 设置后，同步把新规则记入 `ruleSources`（源 `'user'`），保持会话内一致。

### 4.6 headless.ts 接线

同样从 loadLayeredSettings 构造并注入 `ruleSources`/`denySources`（headless 无弹窗，但硬拒绝文本要带来源、行为与 TUI 一致）。

### 4.7 PermissionDialog.tsx 渲染

读 `ask.reason`，在高危警告下方插入一行：

- `{type:'rule', rule.behavior:'deny'}` → `⚠ 命中 deny 规则 {value}（来自 {permissionSourceName(source)}）`（`T.err` 色）。
- `{type:'hook'}` → `权限被 hook {hookName} 拒绝`（`T.dim`）。
- `undefined`（普通 ask）→ 维持现状不渲染。

## 5. 数据流

```
loadLayeredSettings → permissionSources{allow,deny}
   │ allow → ruleSources
   │ deny  → buildDenySourceMap → denySources
   ▼
PermissionContext{ruleSources, denySources, ask, deny, ...}
   ▼
checkPermission
   ├ deny 非Bash → {ok:false, reason含来源, decisionReason} → 工具错误文本
   ├ deny Bash   → pc.ask(.., decisionReason) → PendingAsk.reason → PermissionDialog
   ├ allow rule  → {ok:true, decisionReason}（底座，无展示）
   └ 普通 ask    → pc.ask(.., undefined) → 弹窗维持现状
```

## 6. 测试策略

- `settingsLayers`：permissionSources 单 scope / 多 scope 重复归属 / allow 剥离 / 空配置。
- `deny`：buildDenySourceMap builtin 标注 + config 合并。
- `permissions`：findMatchingRule / findBashMatchingRule（单命令、前缀、复合精确、复合分段）；checkPermission deny 非 Bash 硬拒绝文本含来源 + decisionReason；Bash forceAsk 透传 reason 给 ask；allow attach decisionReason；hook deny；用户拒绝。
- `PermissionDialog`：deny rule 行渲染、hook 行、undefined 不渲染（字符串/快照）。
- **真机冒烟**（碰 TUI）：`npm start`，配一条 user deny `~/.secret/**` + 触发 Bash 读它 → 弹窗显示「命中 deny 规则 ~/.secret/**（来自 用户设置）」；触发 Read 内置私钥路径 → 工具错误文本含「来自 内置规则」。

## 7. 向后兼容与风险

- 所有新增字段可选；`checkPermission` 返回与 `ask` 签名扩展处全为新增可选参/字段，旧调用点编译期需补（permissions/useChat/headless/测试 mock）。
- `LayeredResult.permissionSources` 纯新增，`config.ts`/`configReport.ts` 不受影响。
- allow-path decisionReason 现阶段**无消费方**（用户钦定的完整底座，预铺 /permissions UI）——spec 显式声明，非死代码遗漏。
- 教训沿用 3.9：改写路径/权限的测试务必同步 mock 新写函数 + 确定性 settings，避免真写 `~/.deepcode` 自污染。

## 8. 偏离 CC 记录

- 用文本 reason + 轻量 `PermissionDecisionReason` 联合，不引 CC 的 `PermissionDecisionDebugInfo` 调试视图（deepcode 无 debug 面板）。
- 来源显示名用中文（deepcode 既有弹窗/文本均中文）。
- 复合 Bash 命令收敛为单 `rule`，不引 `subcommandResults`。
