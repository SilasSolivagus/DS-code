# 2.4 Prompt 缓存对齐 — 设计

> master roadmap 第 2 层「运行时/推理」之 2.4。状态：设计已批准（2026-06-18）。

## 背景与范围裁定

CC（Anthropic API）的 prompt caching 靠请求侧显式 `cache_control: {type:"ephemeral"}` 标记 + TTL（5m/1h）+ scope（global/org）+ 动态断点放置（marker 在最后 1~2 条消息）+ break 检测。实读 CC 源码（`services/api/claude.ts`、`utils/modelCost.ts`、`services/api/promptCacheBreakDetection.ts`）确认：**这套标记/TTL/scope/断点逻辑全部依赖 Anthropic API，对 DeepSeek 不适用**——DeepSeek 是自动磁盘前缀缓存，按最长公共前缀命中，无请求侧控制参数。

deepcode 现状盘点（**实质已对齐八成**）：

- 计费/统计已做：`api.ts:8,34` 读 `prompt_cache_hit_tokens`；`pricing.ts` 按 hit/miss 分价（`deepseek-v4-flash`/`deepseek-v4-pro` 单价已核实）；`stats.ts` 聚合命中率；`/cost`、`/stats` 命令展示命中与花费；`useChat.ts:163,250-252` 已算 `cacheHitRate()`（注释自称「DeepSeek 状态行核心指标」）。
- 缓存友好 prompt 已做（且是刻意设计）：`prompt.ts:29` 系统前缀「只在会话启动时调用一次，产物整个会话静态——这是 KV 缓存命中的前提」；`loop.ts:25-28,261-266` reminder「只动最新后缀，不破坏 KV 缓存」，历史 append-only。

**讽刺缺口**：`cacheHitRate()` 被注释为「状态行核心指标」、却从没进过 `StatusFooter` 状态行（Row 2 只有 `Context [bar] N% · $cost`）。

本件只补三块真实增量 + 把隐式缓存纪律固化成回归守卫。**明确不做** CC 的标记/TTL/scope/断点/break 检测/缓存开关（对 DeepSeek 无意义）。

## 组件 1 — savings 纯函数（`src/pricing.ts`，非 TUI）

新增导出纯函数：

```
cacheSavingsUSD(model: string, hitTokens: number): number
  = hitTokens × (p.miss − p.hit) / 1_000_000
  未知模型（PRICES 无此 key）→ 0
```

含义：缓存命中的那部分输入 token，若按 miss（未命中）价计本要多花的钱，即缓存省下的金额。复用现有 `PRICES` 表。

单测（`test/pricing.test.ts`）：
- 已知模型按差价算对（如 flash：hit=0.0028, miss=0.14 → savings = hitTokens×0.1372/1e6）。
- 未知模型 → 0。
- hitTokens=0 → 0。

## 组件 2 — core state 暴露 `cacheSavings()`（`src/tui/useChat.ts`）

在 createChatCore 内新增 `cacheSavings(): number`，累加各 usage record 的 `cacheSavingsUSD(u.model, u.usage.prompt_cache_hit_tokens)`。镜像现有 `cacheHitRate()`/`sessionCost()` 的写法（同一 `usageLog.reduce`），加入 core 返回对象（`useChat.ts:260` 那一行的暴露列表）。headless 不涉及（无状态栏）。

`/resume` 恢复会话后 `usageLog` 已重灌（`useChat.ts:289-290`），`cacheSavings()` 自动正确，无需额外处理。

## 组件 3 — 状态栏渲染（`src/tui/components/StatusFooter.tsx` + `App.tsx` + `FullscreenApp.tsx`，TUI）

`StatusFooter` props 新增 `hitRate: number`（0~1）、`cacheSavings: number`（USD）。

Row 2 渲染规则：
- 有请求（`hitRate > 0` 或更稳妥地由调用方判断 prompt>0；本设计用 `hitRate>0` 作为「有缓存命中」的显示门槛）→
  `Context ▓▓░░░░░░░░ 40% · cache 87% (−$0.0089) · $0.0123`
  - `cache 87%` = `Math.round(hitRate*100)`，dim 色；
  - `(−$0.0089)` = `cacheSavings.toFixed(4)`，dim 色；
- **无缓存命中（`hitRate === 0`，含会话刚开始无请求）→ 隐藏整个 cache 段**，只显示 `Context ▓▓░░ 40% · $0.0123`（与现状一致），避免 `cache 0% (−$0.0000)` 噪音。

`App.tsx`（`StatusFooter` 调用处，约 256 行）与 `FullscreenApp.tsx`（约 278 行）各传 `hitRate={state.cacheHitRate()}` `cacheSavings={state.cacheSavings()}`。

测试（`test/tui.statusfooter.test.tsx`）新增 case：
- `hitRate>0` → Row 2 含 `cache` 与 `−$`；
- `hitRate===0` → Row 2 不含 `cache`，仍含 `Context` 与 `$cost`。

> 注：`StatusFooter` 行数随 memory/tool 行按需变化，App 的 IME 光标偏移按 footer 行数动态计算（见 `StatusFooter.tsx:25` 注释）。本件 cache 段只在 Row 2 内追加文本、不新增行，故不影响 footer 行数/光标偏移。

## 组件 4 — 前缀稳定性回归守卫测试（只加测试，不改产品代码，非 TUI）

把「系统前缀静态 + reminder 只追后缀」的隐式纪律固化成测试，防未来回归把易变内容塞进前缀、击穿 DeepSeek 自动缓存。

1. `test/prompt.test.ts`：`buildSystemPrompt` 同输入连调两次，产物字节相同（会话内确定性/静态）。
2. `test/loop.test.ts`（复用现有脚本化 `chatStream` mock + `drain` + `makeDeps`）：跑一轮含 tool 调用、`deps.reminders` 返回非空数组，断言：
   - (a) reminder 文本只追加到最后一条 `tool` 消息末尾（`<system-reminder>` 块出现在 tail tool 消息）；
   - (b) 本轮 reminder 注入前已存在的消息（system/user/assistant/前面的 tool）内容字节不变——即 reminder 不修改前缀。

## 不做（YAGNI）

- 显式读 `prompt_cache_miss_tokens`：现推导 `miss = prompt − hit` 已正确，savings 也不需要 miss 字段。
- 移植 CC `promptCacheBreakDetection`：DeepSeek 无显式断点，且无机制会意外击穿。
- 缓存开关（`DISABLE_PROMPT_CACHING` 等）：DeepSeek 缓存自动、无法经 API 禁用。
- TTL/scope/`cache_control` 标记：Anthropic 专属。
