## 0. 背景与动机

deepcode 现在只有单一 settings 文件 `~/.deepcode/settings.json`（`config.ts` `loadSettings()` 读一个文件）。3.9 引入 CC 式分层，是 master roadmap 的**隐藏拓扑根**：解锁 3.7「来源层级」、是所有未来项目级配置的安全前置。

安全威胁模型（2026-06-18 安全加固批 `docs/specs/2026-06-18-deepcode-security-threat-model.md`）已点名：**引入项目级/共享 settings 前必须先补 hook SSRF + URL 白名单，否则项目级文件 = 供应链 RCE**。故 3.9 把分层与 SSRF 加固捆绑交付。

## 1. Scope 与优先级

4 层，优先级**低 → 高**（对齐 CC，去掉 enterprise/policy——归第 6 层云管平台）：

| Scope | 路径 | 信任 | 危险字段 |
|---|---|---|---|
| `user` | `~/.deepcode/settings.json`（现有，路径不变） | 可信 | ✅ 全收 |
| `project` | `<cwd>/.deepcode/settings.json`（入库共享） | **不可信** | ❌ 剥离 |
| `local` | `<cwd>/.deepcode/settings.local.json`（应 gitignore） | 可信*（见 §3.2 git 闸门） | ✅ 全收* |
| `flag` | `--settings <path>`（CLI 临时） | 可信 | ✅ 全收 |

- `<cwd>` = 进程启动时的 `process.cwd()`。本期在会话启动时一次性解析；运行中 `cd`（CwdChanged）不重载分层——记为 follow-up（见 §9）。
- 缺失文件 / JSON 解析失败 / 非对象：该 scope 静默跳过（不崩，对齐现 `loadSettings` 的 try/catch 容错），higher scope 照常加载。

## 2. 合并语义

对齐 CC `settingsMergeCustomizer`（settings.ts:538）：

- **数组**：concat + 去重（`uniq`），即 `permissions.allow`/`permissions.deny`/`skills.deny`/`allowedHttpHookUrls`/`httpHookAllowedEnvVars` 跨 scope 累加。
- **对象**：深合并（lodash mergeWith 语义；deepcode 自实现等价合并，避免引 lodash——只对已知 Settings 结构合并）。
- **标量**：高优先级 scope 胜（后者覆盖）。
- 合并顺序：`user → project(已剥离) → local(可信或降级) → flag`。

**剥离发生在「该 scope 原始对象解析后、并入合并前」**（strip-before-merge）。

## 3. 信任边界（安全核心）

### 3.1 危险字段（仅从 `project` 剥离；`user`/`flag` 全收）

剥离集 = 任何能导致 RCE / 凭证外泄 / API MITM / 查询外泄 / 安全策略弱化的字段：

| 字段 | 剥离粒度 | 攻击（不剥的话） |
|---|---|---|
| `apiKey` | 整键 | 凭证替换 / 计费重定向 / 响应篡改 |
| `baseURL` | 整键 | 把每次 LLM 调用重定向到攻击者端点 = 全程 MITM 指令与工具输出 = RCE 等效 |
| `hooks` | 整键 | command hook 跑任意 shell（#1 向量） |
| `mcpServers` | 整键 | 会话启动 spawn 任意进程 |
| `webSearch` | 整键 | 含 bocha/tavily apiKey → 静默换成攻击者搜索账户 = 查询/结果外泄 |
| `allowedHttpHookUrls` | 整键 | 加项 = **放宽** SSRF 白名单（安全弱化） |
| `httpHookAllowedEnvVars` | 整键 | 加项 = 放宽 env 插值白名单（外泄面扩大） |
| `permissions.allow` | **嵌套删除**（保留 `permissions.deny`） | 静默自动批准任意命令 = RCE。`checkPermission` 命中 allow 即 `{ok:true}` 硬放行，比 CC 还危险 |
| `skills.sources` | **嵌套删除**（保留 `skills.deny`/`listingBudgetChars`） | 改 sources 可关掉 `~/.claude/skills` 里用户可信/安全技能，或启用用户特意关的源 |

**⚠️ 实现铁律（opus 专家 Blocker 1）**：`permissions` 与 `skills` 是对象，**不能整键剥**（会连 `deny` 一起丢）；且若不剥、靠合并后过滤，数组 concat customizer 会把 project 的 `allow` 重新拼进来。必须在 **raw project 对象上做精确嵌套删除**：`delete raw.permissions.allow`、`delete raw.skills.sources`，再走 per-field 解析与合并。

**保留集（project 可贡献）**：`compactTokens`、`costWarnCNY`、`maxToolResultChars`、`model`、`inline`、`permissions.deny`、`skills.deny`、`skills.listingBudgetChars`。
- `permissions.deny` 保留理由：deny 早于 allow 求值、只增不删（concat+uniq 无删除原语）→「shadow-an-allow」攻击结构上不可能。残余风险仅低危 DoS（过宽 deny 把仓库读写降级 ask / 硬拒）→ §6 `/config` 必须显示「N 条 deny 来自 project」非静默。
- `model`/`compactTokens`/`maxToolResultChars`/`listingBudgetChars` 为低危：可选 clamp（§9 follow-up，非阻塞）。

### 3.2 `local` git-tracked 信任闸门

`local` 默认可信（对齐 CC：CC 的信任排除函数只排除 projectSettings、从不排除 localSettings），**让用户能在 `settings.local.json` 里设 per-project 的 apiKey/hooks/mcpServers 而不入库**。

但 `.deepcode/settings.local.json` 住在仓库目录、gitignore 仅是建议性 → 恶意 repo 可强提交一份。**闸门**：加载 local 前用 `git ls-files --error-unmatch <path>`（在 `<cwd>`）判断该文件是否被 git 跟踪：
- **未跟踪**（gitignore、你自己写的）→ 可信全收。
- **被跟踪**（= repo 随身携带、非你手写）→ **降级到 project 信任级**（按 §3.1 剥离危险字段），并在 `/config` 标注「settings.local.json 被 git 跟踪，已按不可信来源降级」。
- 非 git 仓库 / git 不可用 → 视为未跟踪（可信）。文件不存在 → 跳过。

降级（而非完全忽略）保留了安全字段的可用性，又关死了 `git clone` 恶意 repo 的攻击向量，且不需要弹框（TUI 攒最后批）。

## 4. SSRF 加固（两层，对齐 CC）

### 4.1 新模块 `src/ssrfGuard.ts`

移植 CC `ssrfGuard.ts`（已实读）：

- `isBlockedAddress(address)`：封 IPv4 `0/8`、`10/8`、`100.64/10`（CGNAT，含阿里云元数据 100.100.100.200）、`169.254/16`（云元数据）、`172.16/12`、`192.168/16`；封 IPv6 `::`、`fc00::/7`、`fe80::/10`、IPv4-mapped-IPv6（提取内嵌 v4 再判，防 `::ffff:a9fe:a9fe` 绕过）；**放行 loopback** `127.0.0.0/8`、`::1`（本地 dev hook 是主用例）。纯函数，TDD 全覆盖（含 mapped-v6 各表示形态）。
- `ssrfGuardedLookup(hostname, options, callback)`：dns.lookup 兼容签名。IP 字面量直接校验；否则 `dnsLookup(all:true)` 后逐个 IP 过 `isBlockedAddress`，任一被封即回错。用作连接时 lookup → **校验的 IP = socket 连接的 IP，无 TOCTOU rebind 窗口**。

### 4.2 URL 白名单 + env 白名单

新 Settings 字段（§5）：
- `allowedHttpHookUrls?: string[]`：`undefined` = 不限制；`[]` = 全禁；非空 = hook.url 须匹配至少一条通配模式（`*` → `.*`，其余正则转义；对齐 CC `urlMatchesPattern`）。跨 scope concat。
- `httpHookAllowedEnvVars?: string[]`：设了则与每个 hook 自身 `allowedEnvVars` **取交集**（对齐 CC execHttpHook.ts:163-167）；未设则仅用 hook 自身列表（现状）。

二者均在 §3.1 危险集（project 不可贡献，防放宽）。

### 4.3 接进 `execHttpHook`（hooks.ts，Blocker：现用全局 fetch 未接守卫）

现 `execHttpHook` 用 `deps.fetch`（全局 fetch，**默认跟随重定向、无法注入 per-connect lookup**）。改造：

1. **I/O 前查 URL 白名单**：`allowedHttpHookUrls !== undefined` 且无匹配 → 直接返回 blocked（不发请求）。
2. **dispatcher 两路**（沿用 deepcode 现有 undici + ProxyAgent 模式，见 api.ts:94/webSearch.ts:13）：
   - 代理激活（`https_proxy/HTTPS_PROXY/http_proxy/HTTP_PROXY` 任一存在 **且** 目标 host 不被 `NO_PROXY` 豁免）→ `ProxyAgent(proxy)`，**跳过 ssrfGuardedLookup**（代理自己做 DNS，对齐 CC `envProxyActive` 分支）。
   - 否则 → `new Agent({ connect: { lookup: ssrfGuardedLookup } })`。
3. **`redirect: 'error'`**（或 undici 等价）防 SSRF-via-redirect（首跳过白名单后 302 跳 169.254.169.254 绕过）。
4. env 插值白名单交集（§4.2）。
5. 改用 undici `fetch`（注入 dispatcher），保留现有 CRLF 消毒 + env 白名单（①c 已做）。`deps.fetch` 测试注入点保留。

镜像 `shouldBypassProxy(host)` 小助手（解析 `NO_PROXY`）。

**诚实记录（opus 专家）**：本机强制 `https_proxy=127.0.0.1:8118` → 代理恒激活 → **网络层守卫在本机实际不生效**；本机真实 SSRF 防护 = URL 白名单 + 代理自身域名策略。文档与威胁模型不得号称「两层常开」。又因 project hook 已在 settings 层全剥，网络层守卫的剩余攻击者仅 user/local/flag（可信）→ 网络层是纵深防御。

## 6. `/config` 只读源追溯命令

纯文本只读（无交互编辑——留 4.5 Config 工具 / 5.6 权限 UI）。渲染：

- **每个顶层 key**：合并值 + 来源标注，如 `model: deepseek-v4-flash  [user]`、`compactTokens: 200000  [project]`。
- **数组合并字段**：`permissions.allow: 3 条  [merged: user+local]`。
- **被剥离的 project 字段**：`⚠ project 的 hooks/mcpServers/apiKey/baseURL/permissions.allow/skills.sources 已忽略（不可信来源）`。
- **project 贡献的 deny**：`permissions.deny: 5 条（其中 2 条来自 project）` 非静默。
- **local 降级**：若 git-tracked 降级，标注。
- **末尾**：实际加载的文件路径列表（每条标 scope + 是否降级/缺失）。
- 敏感值（apiKey）打码显示，不回显明文。

`provenance` 按**顶层 key 粒度**追踪：`Map<topKey, scope | 'merged' | {value, scope}[]>`。

headless：`/config` 不在 headless 注册（无人看）或打印纯文本一次即退——按现 headless 斜杠惯例，默认不注册（与交互命令一致）。

## 7. 写路径修复（Blocker 2：洗白 + saveRule）

一旦 `loadSettings()` 返回**合并**结果，任何 `load→改→saveSettings(merged)` 都会把 project/local/flag 值持久化进 user 文件（权限洗白）。

### 7.1 拆 API（config.ts）

- `loadSettings()` → 改为返回**合并**结果（委托 §8 `loadLayeredSettings().settings`）。所有只读消费者（api.ts/headless.ts/index.ts/useChat 运行时）自动获得分层，零改动。
- 新增 `loadRawUserSettings(): Settings` —— 只读 `~/.deepcode/settings.json` 原始解析（= 现 `loadSettings` 的旧实现，单文件）。
- 新增 `saveRawUserSettings(s: Settings): void` —— 写 user 文件（= 现 `saveSettings` 实现 + chmod 600）。
- **退役/改名** 裸 `saveSettings(merged)`，让任何调用者无法误把合并对象落盘。

### 7.2 审计并迁移全部写者

| 站点 | 现状 | 改 |
|---|---|---|
| `config.ts:189 saveApiKey` | `loadSettings()→改 apiKey→saveSettings` | `loadRawUserSettings()→改→saveRawUserSettings`（Setup hook 的 hadKey 判断同样基于 raw user） |
| `useChat.ts:535 saveRule` | `settings.permissions.allow.push(r); saveSettings(settings)` | raw-user RMW：读 user 原始 → push 规则到 user.permissions.allow → `saveRawUserSettings` → fireConfigChange |
| `useChat.ts:778 /permissions rm` | `settings.permissions.allow.splice(i); saveSettings(settings)` | 对 **user-scope** allow 数组按值删除（见下）→ `saveRawUserSettings` |

`/permissions` 列表/删除语义：写入目标恒为 user scope，故 `/permissions` 列表显示 **user-scope** 规则（保 rm 索引一致）；合并视图（含 project deny、local/flag 规则、来源）归 `/config`。runtime `settings.permissions`（合并）仍喂 `checkPermission` 不变。

> 注：`saveRule` 写 user scope 后，运行时内存 `settings.permissions.allow`（合并）也要同步反映新规则，否则本会话后续不立即生效。实现时在 RMW 后把新规则也 push 进内存合并数组（与现行内存即时生效行为对齐）。

## 8. 模块结构

- 新 `src/settingsLayers.ts`：
  - `scopePaths(cwd)` → 各 scope 文件路径。
  - `loadScope(scope, path, {strip})` → 读文件 → JSON parse（容错）→ 若 strip 则嵌套删除危险字段 → 复用 config.ts 各 `parseXxx` → 部分 Settings。
  - `isGitTracked(path, cwd)` → §3.2 闸门（`git ls-files --error-unmatch`，spawn 容错，非 git/出错→false）。
  - `mergeSettingsLayers(layers)` → §2 合并 + 产出 provenance。
  - `loadLayeredSettings(cwd?, flagPath?)` → `{ settings, provenance, sources }`。
- `config.ts`：保留全部 `parseXxx`（被 loadScope 复用）；`loadSettings` 委托；新增 `loadRawUserSettings`/`saveRawUserSettings`；退役 `saveSettings`。
- 新 `src/ssrfGuard.ts`：§4.1。
- `hooks.ts`：`execHttpHook` 改造（§4.3）。
- `useChat.ts`/`headless.ts`：写者迁移（§7.2）+ `/config` 命令 + flag scope 接线。
- `index.ts`：`--settings <path>` CLI flag 解析 → 透传 flagPath 到 loadLayeredSettings 入口（TUI + headless）。

## 10. 测试策略

纯逻辑为主（免真机冒烟）；安全件按既定教训**派 opus 实跑对抗用例**。

- 合并：优先级覆盖、数组 concat 去重、标量后者胜、缺文件/坏 JSON 跳过。
- 信任边界：project 的 apiKey/baseURL/hooks/mcpServers/webSearch/allowedHttpHookUrls/httpHookAllowedEnvVars 被丢；`permissions.allow` 丢而 `permissions.deny` 留；`skills.sources` 丢而 `skills.deny` 留；嵌套删除不误伤保留键。
- git 闸门：tracked local 降级剥离、untracked local 全收、非 git 仓库可信。
- SSRF：`isBlockedAddress` 全网段 + IPv4-mapped-v6 各形态 + loopback 放行；URL 白名单 undefined/[]/模式；代理激活跳过守卫、NO_PROXY 豁免回到守卫；redirect:error。
- 写路径：saveApiKey/saveRule/permissions rm 均不污染 user 文件（不写入 project/local/flag 值）；rm 索引对 user scope 一致；内存合并即时生效。
- provenance + `/config` 渲染：来源标注、剥离警告、project deny 计数、文件列表、apiKey 打码。
- 回归：现有单文件 `~/.deepcode/settings.json` 行为字节级不变（user scope = 旧路径）。
