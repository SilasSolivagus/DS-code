## 背景

CC 全机制对齐进入 C3 Token/Compact 批，首件 2.5「Token 计数/估算」。

按既定流程派两个专家 agent 实读对照（CC 源码 `/Users/silas/Desktop/src`，deepcode 源码 `/Users/silas/loop/deepcode`），结论与 2.4 高度相似：**CC 2.5 的大部分机器对 DeepSeek 不适用，但有一块 DeepSeek 专属的真金（中文 token 低估）值得做。**

### CC 2.5 核心机制（`services/tokenEstimation.ts`）

- **精确计数** = 调 Anthropic `messages.countTokens()` API 端点（发送前预估，不靠实际响应）。
- **多后端分流**：Bedrock `CountTokensCommand` / Vertex beta 过滤 / Haiku fallback。
- **本地估算 fallback**：`roughTokenCountEstimation = content.length / bytesPerToken`，默认 `bytesPerToken=4`，JSON/JSONL `=2`；按 block 类型分别估（图像固定 2000、tool_use 估 `name+JSON(input)`、thinking 只计思考文本）。
- **消费链**：`tokenCountWithEstimation(messages)` = 上次 API 精确 usage + 新增消息粗估 → `getAutoCompactThreshold(model)`（effective window − 13k buffer）→ 警告（−20k）→ 阻断（−3k）。
- **context window 动态解析**（`getContextWindowForModel`）：优先级 env 覆盖 → `[1m]` 后缀 → 模型 capabilities → 默认 200k；effective window = window − 输出预留（≤20k）。

### deepcode 现状（专家实测）

**已对齐（核心精确）：**
- token 数直接取 DeepSeek API 流式 `chunk.usage`（`src/api.ts:5-9,25,32-34`，字段 `prompt_tokens`/`completion_tokens`/`prompt_cache_hit_tokens`）——**精确**。
- hit/miss 分价正确（`src/pricing.ts:13-17`，`miss = prompt − hit` 推导正确）。
- compact 触发基于真实 `lastPromptTokens`（`src/tui/useChat.ts:730`、`src/compact.ts:42-44`）。
- cacheHitRate/cacheSavings 已上状态栏（2.4 件）。

**真实缺口：**
- **无独立 token 估算模块**，散落两处本地启发式：
  - `formatContext`（`src/commands.ts:29-51`，`/context` 命令）用 `Math.round(n / 4)`。
  - spinner 动画（`src/tui/useChat.ts:627`）用 `Math.ceil(ev.delta.length / 3)`（turn_end 被 API 真值覆盖，仅观感）。
- **中文低估 2-4×**：`chars/4` = 0.25 token/字 是英文基准；DeepSeek 中文实际约 0.6 token/字。这是 DeepSeek 重度中文场景的专属问题，CC 那套完全没有。
- **反应式 compact 触发**：现用上一轮 API 返回的 `prompt_tokens` 判，单轮巨型工具结果可能在下次检查前冲破上下文。
- **context window 硬编码 200k**（`compactTokens` 默认值同时当触发阈值和 `contextPct` 分母，`src/settingsLayers.ts:50`、`src/config.ts:101`、`src/tui/useChat.ts:300`），但 V4 系列实际 1M 上下文。

## 决策（brainstorm 拍板 + opus 专家对抗评审后修订）

1. **范围 = 全做**：中文精度 + 模块化整合 + 发送前预估触发 compact。
2. **估算方式 = 官方比例启发式、零依赖**：CJK ×0.6 + ASCII/拉丁 ×0.3 + 数字/符号按官方「通常算一个 token」归类（见 §权重表）。不引 tokenizer 依赖（与 deepcode 轻量哲学一致）。
3. **context window = 模型感知**：按模型解析真实 window（flash/pro 都 1M，表驱动可扩展）。
4. **阈值 = window − buffer，完全对齐 CC**（用户二次确认坚持）：放弃 200k 扁阈值作为触发点。**已向用户两次明示并接受**默认触发点从 200k → ~（window − buffer ≈ 980k），默认上下文涨到接近 1M、单轮花费/延迟上升的后果。
5. **`compactTokens` 默认值改为 `undefined`（类型改 optional）**——这是让决策 4 真正生效的前提（见 §🔴关键修订1）。语义从「触发阈值」反转为「可选省钱 override」：未设 = 走派生阈值（~980k）；设了 = `min(派生阈值, compactTokens)` 作更紧上限。需连带改 4 处 + Config 工具 + 文档（见 §架构）。
6. **状态栏上下文条改显绝对值** `used / window`（如 `132k / 980k`）而非纯百分比条——因分母变 ~980k 后百分比条几乎永不亮（专家 #5）。详见 §contextPct 改造。
7. **`estimateMessagesTokens` 按 deepcode 扁平 string 结构做**，不搞 block 级累加、不留图像扩展点（专家 #7：deepcode 消息 `content` 永远是 string、`tool_calls.arguments` 已是 string，无 Anthropic block 数组；图像是 YAGNI，V4 纯文本且接口不收图）。
8. **buffer 具体数值留实现阶段核实 DeepSeek max output 再定**。

## 待实现阶段核实的外部事实

- **✅ DeepSeek 官方 token 估算比例（已查实，2026-06-19 web）**：「1 英文字符 ≈ 0.3 token、1 中文字符 ≈ 0.6 token」；官方注「一个中文词/英文词/数字/符号通常算一个 token，不同模型 tokenization 不同、实际以 usage 返回为准」。来源 DeepSeek API Docs `api-docs.deepseek.com/quick_start/token_usage`。→ 权重表见 §架构，数字/符号按「通常算一个 token」处理（实现阶段定具体归类，见 §权重表占位）。
- **DeepSeek V4 context window**：项目史记 + 公开资料「v4 系列 1M 上下文」——实现计划阶段最终确认 flash 与 pro 是否都 1M（模型表 `resolveContextWindow` 用）。
- **DeepSeek max output tokens**：api.ts 现不显式传 `max_tokens`（走服务端默认，`api.ts:116-133` 确认未传）——确认默认上限，用于定输出预留 buffer。
- **buffer 取值**：CC 用输出预留 ≤20k + autocompact 13k。deepcode 取值按上面 max output 核实后定（实现计划锁死具体常量）。

## 架构

### 新模块 `src/tokenEstimate.ts`（对齐 CC 独立 `tokenEstimation.ts`）

纯函数集合，无依赖，全可单测。是本件的核心单元：单一职责（token 估算 + 上下文窗口/阈值解析），通过明确接口被各消费者调用，可独立理解和测试。

```ts
// CJK 感知文本估算：遍历字符按 Unicode 类别加权累加。空串/undefined → 0，绝不抛。
export function estimateTextTokens(s: string): number

// 消息数组估算：按 deepcode 扁平 OpenAI 结构逐条累加（专家 #7）。
//   每条：estimateTextTokens(content ?? '')
//   + 对 assistant 的 tool_calls：Σ estimateTextTokens(fn.name + fn.arguments)
// 无 block 数组、无图像分支（V4 纯文本）。
export function estimateMessagesTokens(messages: any[]): number

// 模型感知 context window：env 覆盖 → 模型表(flash/pro 1M) → 默认。未知模型回落默认，绝不抛。
export function resolveContextWindow(model: string): number

// compact 触发阈值：window − 输出预留 − autocompact buffer。纯函数。
//（compactTokens override 的 min 在调用侧算 effectiveThreshold，不进本函数）
export function computeCompactThreshold(model: string): number
```

**`estimateTextTokens` 权重表**（基于已查实官方比例）：
- CJK 统一表意文字区段（`一-鿿` 等，含中日韩表意） → ×0.6/字
- ASCII / 拉丁字母 → ×0.3/字
- 数字 / 标点 / 符号 → 官方「通常算一个 token」，实现阶段定具体归类（占位方案：按 whitespace 分词后，数字串/符号串各计 ~1 token；或保守按 ×0.3/字。实现计划锁死并配测试断言）
- **实现要点**：单遍扫描按 `codePointAt` 判区段，累加后 `Math.ceil`。

**`resolveContextWindow` 优先级链**（对齐 CC `getContextWindowForModel`）：
1. env 覆盖 `DEEPCODE_MAX_CONTEXT_TOKENS`（命名实现阶段终定）
2. 模型表：`{ 'deepseek-v4-flash': 1_000_000, 'deepseek-v4-pro': 1_000_000 }`，**未知模型（用户 `/model` 切任意 OpenAI 兼容端点）回落默认 200_000**（保守，避免对未知大窗口模型误判不 compact）
3. 默认 `200_000`

**`computeCompactThreshold`** = `resolveContextWindow(model) − 输出预留 − autocompactBuffer`。具体 buffer 常量实现阶段定（依赖 DeepSeek max output 核实）。

### 🔴 关键修订（opus 专家对抗评审 3 个阻断项的落地方案）

#### 修订1：`compactTokens` 默认值 200k → undefined（否则决策 4 永不生效）

专家 #1：现状 `compactTokens` 默认硬编码 `200_000`（`settingsLayers.ts:50`、`config.ts:101`，类型 `config.ts:36` 还是非 optional `number`）。若不改，`effectiveThreshold = min(派生~980k, 200k)` 永远取 200k，「对齐 CC ~980k」根本不生效。

**连带改动清单（实现计划逐项 TDD）**：
- `config.ts:36` 类型 `compactTokens: number` → `compactTokens?: number`
- `config.ts:101` `raw?.compactTokens ?? 200_000` → `raw?.compactTokens`（不再注默认）
- `settingsLayers.ts:50` DEFAULT_SETTINGS 删 `compactTokens: 200_000`
- `settingsLayers.ts:137` 分层合并白名单保留 `compactTokens`（仍可被各层设置）
- `configTool.ts:36` read/write 保留（值可为 undefined，read 返回 undefined 时显示「未设（走派生阈值）」）
- **所有读 `settings.compactTokens` 处**（`useChat.ts:300/658/730`）改走新的 `effectiveThreshold(model, settings.compactTokens)` 辅助（见修订2）
- **向后兼容**：老用户 settings.json 里显式写了 `compactTokens: 200000` 的，仍生效为 200k 省钱上限（min 取 200k），行为不变；没写的，从「隐式 200k」变成「派生 ~980k」——这正是决策 4 要的行为变更。

`effectiveThreshold` 辅助（useChat 侧，非纯函数因读 settings）：
```ts
const effectiveThreshold = (model: string) => {
  const derived = computeCompactThreshold(model)
  return settings.compactTokens != null ? Math.min(derived, settings.compactTokens) : derived
}
```

**新签名**（语义不变，入参换成预估值与派生阈值）：

```ts
//                ^ 调用侧传 estimateNextRequest()      ^ 调用侧传 effectiveThreshold(model)
```

**测试迁移（MEMORY 记的 3.9 教训）**：`test/compact.test.ts` 现有针对旧四参的 `shouldAutoCompact` 用例必须同步迁移（参数含义变但形状同为四参，主要是更新调用点语义注释 + 新增「预估越阈触发」用例）。实现计划列为显式任务，避免改签名漏改测试自污染。

#### 修订3：baseline 不变量（deepcode 专属，明确 5 个同源点 + 防越界 + 含上轮输出）

专家 #3/#4：CC 从 messages 反向扫 usage（自愈），但 deepcode 消息**不存 usage**（usage 只在 turn_end 事件→usageLog，不落 messages），照搬不了。改**手动 baseline**，但必须解决专家指出的脆弱点。

**不变量：`baselineLen` 与每次 `lastPromptTokens` 写入原子配对**（实读确认 `lastPromptTokens` 共 5 个写入点）：

| 位置 | `lastPromptTokens` 写入 | 配对 `baselineLen` | 理由 |
|---|---|---|---|
| `useChat.ts:270` init | `= 0` | `= 0` | 空会话，下次发送全估 |
| `useChat.ts:339` clear/newSession | `= 0` | `= 0` | 重置，全估新 messages |
| `useChat.ts:500` compact | `= 0` | `= 0` | rebuilt 全是新上下文，全估 |
| `useChat.ts:643` turn_end | `= ev.usage.prompt_tokens` | `= messages.length`（**捕获点见下** ） | 精确基线对应当前消息集 |
| `useChat.ts:857` resume | `= 0`（现状） | `= 0` | 保守：恢复后首次发送全估（不信任跨进程游标） |

**发送前预估**：
```ts
const estimateNextRequest = () =>
  lastPromptTokens + estimateMessagesTokens(messages.slice(Math.min(baselineLen, messages.length)))
//                                                       ^ 防越界 clamp（survive rewind/截断）
```

**两个正确性要求（实现计划必须 TDD 钉死）**：
1. **增量必须含上一轮 assistant 输出**（专家 #4）：`prompt_tokens` 是上轮**输入**、不含上轮模型刚生成的 assistant 回复；该回复在下轮才进 context。所以 `baselineLen` 捕获点必须落在「上轮 assistant 输出 push 进 messages **之前**对应的长度」，使该输出落进下次 slice。→ **实现计划须先读 `loop.ts` 确认 turn_end 事件与 assistant 消息 push 进共享 `messages` 的先后**，并写测试：「一轮产出 N 输出 token → 下次发送前预估约升 N」。
2. **越界防御**：`Math.min(baselineLen, messages.length)` 兜底 rewind/截断把 messages 改短的情形。

> 说明：选「全部归零点 baselineLen=0 → 下次全量估」是有意的保守，牺牲一点 CPU（重估整个 messages）换绝对不会越界/漏算。compact/clear/resume 后第一次发送本就该全量重估（上下文刚重置）。

### 发送前预估触发 compact（对齐 CC `tokenCountWithEstimation`）

现状反应式（用上轮 `prompt_tokens`）→ 改 **精确基线 + 增量估算**。完整不变量、5 个同源点、防越界 clamp、「增量含上轮 assistant 输出」的正确性要求见上方 **§🔴关键修订3**（不在此重复）。核心：

### contextPct 改造 + 状态栏改显绝对值（专家 #5）

**问题**：分母从 `compactTokens`(200k) → `effectiveThreshold(model)`(~980k) 后，典型会话几万~十几万 token / 980k = 个位数%，`StatusFooter.tsx:30` 的 10 格条 `round(pct/10)` 基本永远 0-1 格、80%/95% 变色阈值永不触达 → 上下文条形同虚设。

**决策（用户选）**：状态栏上下文段改**显绝对值 + 真实窗口** `used / window`，如 `132k / 980k`：
- `contextPct()` 仍保留（分母改 `effectiveThreshold(model)`，供需要百分比的旧接口/警告逻辑），但**状态栏主显改成绝对值文本**。
- 新增 `contextUsed(): number`（= `lastPromptTokens`）+ `contextWindow(): number`（= `effectiveThreshold(model)`），StatusFooter 渲染 `${fmtK(used)} / ${fmtK(window)}`（`fmtK` 复用现有数字格式化或新增小工具，千分位/k 缩写）。
- 颜色：用 `used/window` 比值套现有 `contextBarColor` 阈值（≥80% 黄、≥95% 红）——绝对值大窗口下仍能在接近 compact 时变色，比纯条子有意义。
- `useChat.ts:658` 的 ≥90% 一次性警告：分母同步改 `effectiveThreshold(model)`。

## 数据流

```
DeepSeek API stream chunk.usage (精确)
  → useChat turn_end(:643): lastPromptTokens = prompt_tokens；baselineLen = messages.length（原子配对，§修订3）
  → 下轮发送前(:730):
      预估 = lastPromptTokens + estimateMessagesTokens(messages.slice(min(baselineLen, len)))
      effectiveThreshold = compactTokens!=null ? min(computeCompactThreshold(model), compactTokens) : computeCompactThreshold(model)
      shouldAutoCompact(预估, effectiveThreshold, failures, max) ? → doCompact('auto')（compact 内 lastPromptTokens=0, baselineLen=0）
  → 状态栏: used = lastPromptTokens, window = effectiveThreshold(model) → 显 "used / window" + 比值变色
           contextPct() = min(100, round(lastPromptTokens / effectiveThreshold × 100))（保留，供警告逻辑）

本地估算（无 API 数据时）:
  formatContext / spinner → estimateTextTokens (CJK 感知)
```

## 错误处理 / fail-safe

- `estimateTextTokens` 对空串/undefined 返回 0，绝不抛。
- `resolveContextWindow` 未知模型回落默认值，绝不抛。
- 发送前预估只决定「要不要提前 compact」；估偏不致命（compact 失败有现成熔断 `MAX_AUTO_COMPACT_FAILURES`）。预估是纵深，API 真值仍是事后真相来源。
- compactTokens override 缺省（undefined）时不参与 min（即纯走派生阈值）——见 §修订1，默认值已改 undefined 使其成立。

## 明确砍掉（YAGNI / N/A，同 2.4 砍 Anthropic 专属）

- ❌ `countTokens` API 端点（DeepSeek 无此能力）。
- ❌ Bedrock `CountTokensCommand` / Vertex beta 过滤 / Haiku fallback（单后端）。
- ❌ 引 tokenizer 依赖（启发式足够驱动 compact 决策，±误差可接受）。
- ❌ **block 级估算 / 图像分支 / 扩展点**（专家 #7：deepcode 消息是扁平 OpenAI string 结构，无 Anthropic block 数组；V4 纯文本且接口不收图。`estimateMessagesTokens` 直接对 `content` string + `tool_calls.arguments` string 估算，不搭 block 骨架、不留图像扩展点）。
- ❌ 补 `prompt_cache_miss_tokens` / `cache_creation_input_tokens`（`miss = prompt − hit` 推导已正确，同 2.4 结论）。
- ❌ CC 的三档警告/阻断缓冲（warning −20k / error −20k / blocking −3k）整套——deepcode 现有 ≥90% 一次性提示（`useChat.ts:658`）够用，不引三档。

## 测试

- `estimateTextTokens`：纯中文 / 纯英文 / 中英混排 / 代码片段 各一组，断言落在 ±误差带（真实 token 的 ±15-20%）。空串/undefined → 0。
- `resolveContextWindow`：flash/pro → 1M，未知模型 → 默认 200k，env 覆盖优先。
- `computeCompactThreshold`：window − buffer 计算正确。
- `effectiveThreshold`（useChat 辅助）：**compactTokens=undefined → 走派生 ~980k**（修订1 核心回归，防默认值不生效）；compactTokens=200k → min 取 200k（老用户向后兼容）；compactTokens 大于派生 → 仍取派生。
- **`shouldAutoCompact` 迁移**：现有四参用例语义更新；新增「估算值 > 阈值 && failures<max → true」「估算 ≤ 阈值 → false」「failures≥max 熔断 → false」。
- **发送前预估 + baseline 不变量**（修订3 核心）：
  - 「上次精确小 + 大增量新消息」越阈 → 触发 compact；增量未越阈 → 不触发。
  - **「一轮产出 N 输出 token → 下次预估约升 N」**（钉死「增量含上轮 assistant 输出」，专家 #4）。
  - **baseline 越界防御**：messages 被改短（模拟 rewind/截断）后 `slice(min(baselineLen,len))` 不越界、不抛。
  - **同源重置**：compact/clear/resume 后 baselineLen 归 0、下次全量重估。
- contextPct + 状态栏：分母随模型变；状态栏显 `used / window` 绝对值文本；比值 ≥80%/≥95% 变色。
- 消费者：formatContext 中文样例 token 数显著高于旧 `chars/4`（量化中文不再低估）。
- **配置向后兼容**：老 settings.json 显式 `compactTokens:200000` 仍生效；缺该键的走派生阈值；Config 工具 read undefined 时友好显示。
