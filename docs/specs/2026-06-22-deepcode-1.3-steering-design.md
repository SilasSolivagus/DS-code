# deepcode 1.3 Steering（用户中途转向）设计

- 日期：2026-06-22
- 状态：设计已批准，待 writing-plans
- 对齐对象：Claude Code（CC）steering 机制（L-043）
- 前置调研：本会话两份 agent 报告（CC 源码实读 + deepcode 现状摸底）

## 1. 背景与目标

**Steering** = 用户在模型还在跑当前 turn（流式生成中 / 工具执行中）时插入新消息，**不是硬中断取消，而是"转向"**：把新输入排队或立即软中断重定向，让模型据此调整而非重启。

### CC 的真实机制（实读结论）
- 优先级队列 `now / next / later`（`messageQueueManager.ts`）。
- 中途输入立即入队、不阻塞 TUI 输入框。
- `next/later`：在 **tool_result 边界** 消费，注入成 `queued_command` **attachment**（背景信息，模型自决怎么响应）。
- `now`：`abort('interrupt')` 软中断当前 turn——**与 ESC 共享同一条 AbortController 信号路径**，区别只在 abort 之后的善后：
  - `reason='interrupt'`：保留已生成进度 + 注入 steering 续跑，**不**生成"[已中断]"标记。
  - `reason='user-cancel'`（ESC）：丢弃流式 + 生成 `[Request interrupted by user]` 标记、停。
- attachment 系统是 Anthropic API 专属、**不可移植**。

### deepcode 现状（摸底结论）
- 中途输入**当前被直接丢弃**：`InputBox.tsx:64` busy 时吞掉 Enter（draft 累积但永不提交）。
- 约 40% 基础已具备：
  - `drainInjections` / `injectionBuffer`（Skills 件引入）——在 `loop.ts:296-298` 的 **tool_result 边界**注入 user 消息。**正是 CC 的 steering 注入边界。**
  - `reminders`（`loop.ts:290-294`，system-reminder 后缀）。
  - 后台 task-notification 队列（`tasks.ts`）。
  - `AbortController`（`useChat.ts` 每轮新建，`ctx.signal` 穿到 loop 与 tool）。
- 缺口（~60%）：TUI 捕获中途输入、独立 steering 队列、loop 内 mid-turn 消费点与 interrupt 善后、TUI↔loop 时序同步。

**判断**：与 2.4/2.5「已八成」不同，steering 的核心捕获/重定向路径是真缺的，但注入基础设施（drainInjections → tool_result 边界 user 消息）是理想载体，可大量复用。

### 用户钦定抉择（brainstorm 四问）
1. **Scope**：三档 `now/next/later` 全做（含最复杂的 `now` 软中断）。
2. **注入角色**：user 消息 + `<queued-user-message>` steering 标记（复用 drainInjections 通道，语义贴 CC attachment）。
3. **TUI 交互**：busy 时 Enter→排队 `next`，Alt+Enter→`now`；输入框下方列队 + 档位标记；ESC 拉回最后一条未消费项编辑。
4. **now 中 in-flight tool**：对齐 CC——复用现有 abort 信号路径给 tool 发信号，靠 `signal.reason` 区分 `interrupt`/`user-cancel`。

## 2. 架构与组件

| 组件 | 职责 | 性质 |
|---|---|---|
| **新 `src/steering.ts`** | 会话级优先级队列。镜像 CC `messageQueueManager` 剪到 deepcode 实需。 | 纯逻辑，独立可测 |
| **`src/loop.ts`** | tool_result 边界新增 drainSteering（与 drainInjections 同站点）；两个 abort 检查点新增 `reason==='interrupt'` 软中断分支。 | 改造 |
| **`src/tui/useChat.ts`** | abort 带 reason；持有 steering 队列实例；`ctx.resetSignal`；接线 Enter/Alt+Enter/ESC；订阅列队显示。 | 接线 |
| **`src/tui/components/InputBox.tsx`** | busy 时 Enter→`enqueue('next')`、Alt+Enter→`enqueue('now')`+`onSteerNow`、ESC→有队列则 `popLast` 拉回否则 onInterrupt。 | 改造 |
| **新 TUI 列队展示** | 输入框下方列出已排队消息 + `next`/`now` 标记。可内联进 InputBox 渲染或独立小组件。 | 新增（碰 TUI） |
| **`src/types.ts`** | `ToolContext` / `LoopDeps` 加 steering drain + `resetSignal` 契约。 | 类型 |

### 2.1 `src/steering.ts`（队列）

```ts
type SteeringPriority = 'now' | 'next' | 'later'
interface SteeringItem { id: string; value: string; priority: SteeringPriority }

class SteeringQueue {
  enqueue(value: string, priority: SteeringPriority): SteeringItem  // push 队尾 + notify
  // 取出所有 priority ≤ threshold 的项（FIFO，同优先级保序），并从队列移除
  drainByMaxPriority(threshold: SteeringPriority): SteeringItem[]
  popLast(): SteeringItem | undefined   // ESC 拉回最后一条未消费项编辑
  peek(): SteeringItem[]                // TUI 列队展示用快照
  clear(): void                         // turn 收尾/会话清理
  subscribe(fn: () => void): () => void // TUI 订阅（deepcode 自己的轻量订阅，非 useSyncExternalStore）
}
```

- 优先级权重 `now(0) < next(1) < later(2)`；`drainByMaxPriority('next')` 取 now+next，`'later'` 取全部。
- `id` 用递增计数器 + 前缀（避免 `Math.random`；同 taskList 风格）。**注意**：构造期不可用 `Date.now()`/`Math.random()`。
- 纯内存、不落盘（steering 是会话内瞬态控制指令，无 resume 需求）。

### 2.2 注入格式

每条 steering 项 drain 后包成：

```
<queued-user-message>
用户在你执行过程中补充了这条消息（在你看到它之前已发出）。请据此调整当前工作：
{value}
</queued-user-message>
```

- 作 `{ role: 'user', content: ... }` 追加（复用 loop.ts:296 的 user 注入路径）。
- 与 inline skill 注入**共用 tool_result 边界**，但走独立 drain 函数（`deps.drainSteering`），格式不同。

## 3. 数据流

### 3.1 next / later（软排队）
```
busy 时 Enter → InputBox onSubmit（busy 分支）→ useChat enqueue('next')
  → SteeringQueue（TUI 列队实时显示）
  → [当前 tool 跑完，tool_result 回灌后] loop.ts:298 之后
  → deps.drainSteering('next') → 每条包 <queued-user-message> → push user 消息
  → 下一轮 LLM 看到、自决响应
```
- `later` 仅在"模型本轮无工具调用 + 无其它续跑信号"的自然结束点之前 drain（对齐 CC `sleepRan ? 'later'`），deepcode 无 Sleep 工具，**`later` 暂等同 next 在 tool 边界 drain**；保留档位以备将来（YAGNI 不单独造 later-only 边界，spec 记此偏离）。

### 3.2 now（软中断）
```
busy 时 Alt+Enter → InputBox（key.return && key.meta）→ useChat onSteerNow
  → enqueue('now') + abort.abort('interrupt')
  → 当前 chatStream / in-flight tool 被信号切断
  → loop 捕获，查 signal.reason === 'interrupt'：
      ├─ 不 return 'aborted'
      ├─ 保留已生成 partial assistant 内容（见 §4.2）
      ├─ ctx.resetSignal()（旧 signal 永久 aborted，重建 controller）
      ├─ drainSteering('now') → push <queued-user-message> user 消息
      └─ continue → 下一轮 LLM（用新 signal）
```

## 4. 关键机制：abort reason 区分与 loop 改造（全件最高风险）

### 4.1 reason 复用原生 `signal.reason`（零新类型）
- ESC（`useChat` interrupt）：`abort.abort('user-cancel')`。
- now steering：`abort.abort('interrupt')`。
- loop 两处 abort 检查点改为：先看 `signal.aborted`，再按 `signal.reason` 二分。

```ts
// 善后判定 helper（loop.ts 内）
const isInterrupt = deps.ctx.signal.reason === 'interrupt'
```

### 4.2 两个 abort 检查点的改造

**点 A：流式 catch（`loop.ts:178-192`）**——mid-stream 被切：
- 当前：`signal.aborted` → seal + return 'aborted'。
- 新：
  - `reason === 'interrupt'`：
    - 若 while 循环已累积流式文本（新增 `let streamedText` 累计 text delta，非 reasoning），push `{ role:'assistant', content: streamedText || null }` 保留进度（无 tool_calls）。
    - `ctx.resetSignal()` → `drainSteering('now')` push user 消息 → `continue`（进入下一轮 for，line 164 重读新 `ctx.signal`）。
  - 否则（user-cancel 或无 reason）：维持现状 seal + return 'aborted'。
  - 非中断异常：维持现状（StopFailure hook + rethrow）。

**点 B：turn_end 后（`loop.ts:300-303`）**——mid-tool 被切（assistant+tool_calls 已于 line 194 提交、tool 结果已于 286 回灌，含 line 278 对未执行 rw tool 的"已被用户中断，未执行"标记）：
- 当前：`signal.aborted` → seal + return 'aborted'。
- 新：
  - `reason === 'interrupt'`：`ctx.resetSignal()` → `drainSteering('now')` push user 消息 → `continue`（partial 已天然保留：assistant+tool 结果都在 messages 里）。
  - 否则：维持现状 seal + return 'aborted'。

### 4.3 `ctx.resetSignal`（信号重建）
- 现状：`useChat` 每轮新建 AbortController，`ctx.signal` 指向其 signal。一旦 `abort()`，signal 永久 aborted。
- 新增 `ToolContext.resetSignal?: () => void`：useChat 实现为"新建 AbortController 并更新 ctx 持有的 signal getter"。
- loop 在 §4.2 两分支 continue 前调用 `deps.ctx.resetSignal?.()`；line 164 下一轮 `signal: deps.ctx.signal` 自动读到新 signal。
- **要点**：`ctx.signal` 必须是每轮读取的活引用（loop.ts:164 已在 for 内读取，满足）。useChat 须保证 resetSignal 后 `ctx.signal` 返回新 controller 的 signal（getter 或可变字段）。

### 4.4 in-flight tool 的中断（对齐 CC）
- now 的 `abort('interrupt')` 信号经 `ctx.signal` 已穿到 tool（execCall → tool）。长跑 tool（Bash 等）在自己检查点协作式中止；原子同步写（Write/Edit）无中途窗口、不受影响。
- 这与 ESC 今天的行为同机制，**不新增 ESC 没有的损坏面**。

## 5. TUI 交互

### 5.1 InputBox 改造（`InputBox.tsx`）
- **Enter（busy）**：line 64 不再无条件吞。新逻辑：
  - `key.return && key.meta` → Alt+Enter → `props.onSteerNow(full)`（busy 且非空时）。
  - `key.return`（无 meta，busy）→ `props.onSteerNext(full)`。
  - 非 busy → 维持现状 `props.onSubmit(full)`。
  - 提交后清空输入框（同现状），消息进队列由 useChat 处理。
- **ESC（busy，line 52）**：
  - 队列非空 → `props.onSteerPop()`（拉回最后一条未消费项填入输入框，经 valueOverride/nonce 机制）。
  - 队列空 → 维持现状 `props.onInterrupt()`（硬中断，user-cancel）。
- **modifier 注意**：ink 中 Alt=`key.meta`；line 97 `if (key.ctrl || key.meta || key.tab) return` 在 return-key 块（62-81）**之后**，故 Alt+Enter 会先进 return 块——需在 return 块内分流 meta。Alt+Enter 在不同终端的可达性**留真机冒烟验**；若不可达，回退方案在冒烟阶段定（如改 Shift+Enter 或 `/steer` 命令），spec 记此为冒烟决策点。

### 5.2 列队展示
- 输入框下方（`pending` 续行提示附近）渲染已排队项快照：每条一行 `[next] {截断预览}` / `[now] {…}`。
- 订阅 SteeringQueue 变化重渲染。
- 队列空时不渲染（防闪烁）。

### 5.3 useChat 接线
- 持有 `SteeringQueue` 实例（会话级，dispose 时 clear）。
- `onSteerNext(text)`：`queue.enqueue(text, 'next')`。
- `onSteerNow(text)`：`queue.enqueue(text, 'now')` + `abort.abort('interrupt')`。
- `onSteerPop()`：`queue.popLast()` → 经现有 valueOverride/nonce 机制回填输入框。
- `resetSignal()`：新建 AbortController，更新 ctx.signal 源。
- ESC `interrupt()`（line 1016-1021）：改为 `abort.abort('user-cancel')`。
- 把 `drainSteering` 经 `LoopDeps` 传入 loop（同 drainInjections 接线）。

## 6. 范围边界（YAGNI）

### 砍掉 / 偏离（记录）
- **headless**：无交互输入源，steering UI 不适用；headless 保持现有 drainInjections 不变，不接 steering 队列。
- CC 的 **attachment** 系统（API 专属）→ 改 `<queued-user-message>` user 消息。
- **agentId 子代理隔离**的 steering（CC 有）→ 不做，steering 只作用主会话主循环。
- **消息选择器 / 历史 restore**（CC 长按 ESC）→ 不做。
- **`useSyncExternalStore`** → 用 deepcode 自己的轻量 subscribe。
- **`later` 专用 drain 边界** → 暂等同 next 在 tool 边界 drain（deepcode 无 Sleep，无独立 later 时机）；保留档位枚举以备将来。
- **落盘 / resume** → 不做（瞬态控制指令）。

### 保留接口以备将来
- 优先级三档枚举全留（即便 later 暂同 next）。
- `resetSignal` 是通用的"turn 内信号重建"原语，将来其它软中断场景可复用。

## 7. 测试策略

- **`src/steering.ts` 单测**（纯逻辑，免冒烟）：enqueue 保序、drainByMaxPriority 各阈值正确取并移除、popLast、clear、subscribe 通知、id 唯一。
- **loop interrupt 分支单测**：
  - 点 A：mid-stream `reason='interrupt'` → 保留 partial assistant + 注入 steering + 不 return（续跑）；`reason='user-cancel'` → seal+return 'aborted'。
  - 点 B：mid-tool `reason='interrupt'` → tool 结果committed + 注入 steering + 续跑；`user-cancel` → return 'aborted'。
  - resetSignal 被调用、下一轮读到新 signal。
- **注入格式单测**：`<queued-user-message>` wrap 正确、与 drainInjections 不串。
- **TUI 真机冒烟**（属 TUI 提前破例，同 3.5/5.6）：busy 时 Enter 排队（列队显示）、Alt+Enter now 软中断（看到模型转向、进度保留）、ESC 拉回编辑、ESC 空队列硬中断。Alt+Enter 可达性是冒烟重点。

## 8. 风险与终审重点

- **最高风险 = §4 abort-reason 善后 + resetSignal + partial 保留**：留 opus 全分支终审重点对抗（构造 mid-stream/mid-tool 各 reason 组合、resetSignal 后信号正确性、连发多条 now、interrupt 与 user-cancel 边界）。
- **Alt+Enter 终端可达性**：真机冒烟验，留回退预案。
- **时序**：TUI enqueue 与 loop drain 跨 async 边界——queue 是共享实例（useChat 持有、经 deps 传 loop），无需额外同步原语，但需确认 enqueue 在 loop 下次 drain 前可见（共享引用满足）。
