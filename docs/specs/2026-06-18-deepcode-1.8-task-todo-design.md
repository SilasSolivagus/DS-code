## 0. 关键背景：两套「task」系统别混

CC 有两套互不相干的 task 系统：
- **System A · todo 数据模型**：`TaskCreate/TaskGet/TaskUpdate/TaskList`——磁盘 JSON 待办项（`pending/in_progress/completed`），**不 spawn 任何进程**，是 TodoWrite 的 V2 替代。**← 本件做这个。**
- **System B · 后台进程**：`TaskStop`/`TaskOutput`——杀/读运行中的子进程。deepcode 现有的 `tasks.ts` + `TaskList/TaskOutput/TaskStop` 工具属于这套。

roadmap 原写「基础设施已有 tasks.ts，未暴露成工具」是**概念错位**（把两套混了）。本件**新建** todo 数据模型（System A），与现有后台进程系统（System B）并存。

**唯一撞名**：deepcode 现有 `TaskList`（列后台进程）vs CC 的 `TaskList`（列 todo）。解法：CC 的 `TaskList` 归 todo，现有后台进程列表工具**重命名 `BgTaskList`**（TaskOutput/TaskStop 同义同名不动）。

## 1. 目标与范围

把 CC 的 todo Task 模型忠实搬到 deepcode，**替换**现有 TodoWrite，作为可寻址、落盘、跨 resume 存活的待办系统，并为将来 1.6 多 agent 打底。

### 做（本件范围）
- `Task` 数据模型 + 单调整数 id + 三态 status + `activeForm?` + `metadata?`。
- 落盘：**每任务一文件** `~/.deepcode/task-lists/<sessionId>/<id>.json`，跨 `--continue`/resume 存活。
- `TaskListStore`（新 `src/taskList.ts`）：内存任务 + 走神检测计数 + 落盘 I/O。替换 `src/todo.ts` 的 `TodoStore`。

- `TaskCreate`/`TaskUpdate→completed` 触发 `TaskCreated`/`TaskCompleted` hook（**await 且阻塞**，对齐 CC）。

### 不做（YAGNI / 留增量）

## 2. 数据模型（`src/taskList.ts`）

```ts
export interface Task {
  id: string            // 单调递增整数串 "1","2"…（对齐 CC）
  subject: string       // 简短标题
  description: string   // 要做什么
  status: 'pending' | 'in_progress' | 'completed'
  activeForm?: string   // 进行时文案（如 "Running tests"）；显示在 reminder/TaskList 文本，未来 TUI spinner 用
  metadata?: Record<string, unknown>  // 任意元数据；metadata._internal===true 的任务在 TaskList 中隐藏（对齐 CC）
}
```

- 无 `failed`/`killed`（那是后台进程 System B 的状态）。
- 无 `blocks`/`blockedBy`/`owner`（留 1.6）。
- `status:'deleted'` 不是持久状态，仅作 `TaskUpdate` 的指令值——见 §3 软删除。

## 3. 持久化（每任务一文件，对齐 CC）

**目录**：`~/.deepcode/task-lists/<sessionId>/`，每任务一个 `<id>.json`（与后台任务日志 `~/.deepcode/tasks/` 分开）。常量 `TASK_LISTS_DIR = ~/.deepcode/task-lists`（config.ts，与 `TASKS_DIR` 并列）。

**must-fix（专家审查锁定）：**
1. **原子写**：写 `<id>.json.tmp` 再 `fs.renameSync` 到 `<id>.json`（同 fs 原子）。绝不直接 partial-write 真文件。
2. **加载容错**：`loadFromDisk` 逐文件 `try/catch`，坏 JSON **静默跳过**（镜像 `cleanupOldTaskLogs` tasks.ts:204 与 useChat fileState mtime-skip 风格），单坏文件不毁整列表。
3. **ID 单调性（最高数据丢失风险）**：`loadFromDisk` 后 `nextId = max(已加载数字 id) + 1`。否则 resume 后新建复用 "1" 会覆写旧任务文件。空目录→`nextId=1`。
4. **软删除**：`status:'deleted'` → 标记任务在内存中移除出活跃列表 **但保留磁盘文件**（CC 语义：deleted 任务 TaskGet 仍可查；且保 id 单调正确——max 仍含已删 id）。实现：内存 `Map` 里给该任务打 `_deleted` 内部标记（不写进对外 Task 形态），`list()` 和默认遍历跳过，`get()` 仍能取到。落盘文件保留（含一个 `deleted:true` 字段以便 reload 后恢复删除态）。
5. **写时机**：`create`/`update`/软删 同步 `writeFileSync`+rename 于工具 `call()` 内（与 tasks.ts/config.ts 同步 fs 风格一致，单 agent 小文件够快）。

**会话目录清理（nice-to-have，记为有意延迟）**：`~/.deepcode/task-lists/<sessionId>/` 会随会话累积。启动时可加 `cleanupOldTaskLists`（镜像 `cleanupOldTaskLogs` tasks.ts:194，mtime > 7 天的 dir 删除）。本件**可做可不做**，至少在此记录该泄漏为有意延迟，不算疏漏。

## 4. `TaskListStore`（`src/taskList.ts`，替换 `TodoStore`）

**单一 store 同时持有任务数据 + 走神计数**（专家核实：计数器与「本轮是否改动清单」单一信号强耦合，拆开会引入「忘记 poke 计数器」的 bug；30 行的内聚类，拆分是过度设计）。但**磁盘 I/O 与计数器解耦**：持久化方法只碰任务数据，绝不碰计数器。

```ts
export class TaskListStore {
  private tasks = new Map<string, Task & { _deleted?: boolean }>()
  private nextId = 1
  private dir?: string          // ~/.deepcode/task-lists/<sessionId>；未绑定 sessionId 时纯内存
  private lastUpdateTurn = 0    // 走神计数（绝不落盘）
  private currentTurn = 0

  // bind = 单一(重)初始化入口：(重)指向 session dir、加载磁盘任务、seed nextId、计数器清零。
  // baseDir 可注入便测（默认 TASK_LISTS_DIR）。startup/resume//clear 都调它绑当前 sessionId。
  bind(sessionId: string, baseDir?: string): void
  create(input: { subject; description; activeForm?; metadata? }): Task   // 分配 id、persist、bump lastUpdateTurn
  get(id: string): Task | undefined        // 含已软删的（TaskGet 可查）
  update(id, patch): { ok: boolean; updatedFields: string[] }  // 见 §5 TaskUpdate；status:deleted 软删；bump lastUpdateTurn
  list(): Task[]                            // 排除 _deleted 与 metadata._internal===true
  tick(): void                              // currentTurn++
  staleReminder(): string | null           // 走神提醒（节奏同现状）

  // 私有持久化（只碰任务数据，不碰计数器）
  private loadFromDisk(): void              // 读 dir 下所有 *.json，逐文件 try/catch 跳过坏的；seed nextId=max+1；恢复 deleted 标记
  private persist(id: string): void         // 写 <id>.json.tmp → rename（含软删任务，带 deleted:true）
}
```

> **无独立 `reset()`**：deepcode 的 `/clear` 会创建新 session（新 sessionId→新空目录），resume 绑被恢复会话——故「(重)绑当前 sessionId」即覆盖所有重置场景，`bind` 是唯一入口，不留 reset() 死代码。纯内存单测可 `bind(tmpId, tmpBaseDir)`。

**走神计数器语义（must-fix）：**
- **计数器绝不落盘**。`bind()` 把 `currentTurn=lastUpdateTurn=0` + 加载磁盘任务。
- resume 后：任务从磁盘 reload，但计数器从 0 起 → 首个 post-resume 轮 delta 从 0 长，给正常 3 轮宽限（正确：刚 resume 不该立刻刷屏）。
- **绝不**试图跨 resume「保留 staleness」（轮号跨进程无意义，持久化会致瞬间刷屏或永不提醒）。

**走神检测逻辑**（迁移自 todo.ts:25-29，语义不变，文案改）：
```ts
staleReminder(): string | null {
  const open = this.list().filter(t => t.status !== 'completed')
  const delta = this.currentTurn - this.lastUpdateTurn
  if (!open.length || delta < 3 || delta % 3 !== 0) return null
  return `任务清单已 ${delta} 轮未更新。未完成项：\n` +
    open.map(t => `- [${t.status}] #${t.id} ${t.subject}${t.activeForm ? `（${t.activeForm}）` : ''}`).join('\n') +
    `\n请对照清单检查进度，完成一项就用 TaskUpdate 把它标 completed 并把下一项标 in_progress；计划变了就更新清单。`
}
```

## 5. 工具（`src/tools/taskListTools.ts`）

四个工具。`TaskCreate`/`TaskUpdate` 写盘 → `isReadOnly:false`（串行，防同轮并发写竞态）；`TaskGet`/`TaskList` → `isReadOnly:true`。全部 `needsPermission:()=>false`（CC 把 task 管理列为元数据级自动允许）。全部经 `ctx.taskList` 取 store，缺失时返回错误串（兜底，不应发生于主会话/headless）。

### TaskCreate
```ts
input: { subject: string; description: string; activeForm?: string; metadata?: Record<string, unknown> }
```
- 描述：`创建一个任务到任务清单。多步任务（3 步以上）开始时先建任务列出计划。`
- 行为：`store.create(...)` 分配 id（status 默认 `pending`）→ **触发 `TaskCreated` hook（await 阻塞）**：若 hook block → **删除刚建的任务**并返回错误串（对齐 CC）。否则返回 `已创建任务 #<id>：<subject>`。
- hook payload：`{ hook_event_name:'TaskCreated', task_kind:'todo', task_id:id, subject, task_description:description }`。

### TaskGet
```ts
input: { taskId: string }
```
- 描述：`按 id 取一个任务的全部字段。`
- 行为：`store.get(taskId)` → 有则返回全字段文本（id/subject/description/status/activeForm/metadata），无则 `任务 <taskId> 不存在`。`isReadOnly:true`。

### TaskUpdate
```ts
input: {
  taskId: string
  subject?: string
  description?: string
  activeForm?: string
  status?: 'pending' | 'in_progress' | 'completed' | 'deleted'
  metadata?: Record<string, unknown>   // 合并；值设 null 删该键（对齐 CC）
}
```
- 描述：`更新一个任务。完成一项就把它标 completed 并把下一项标 in_progress；同一时刻至多一项 in_progress。`
- 行为：
  - 任务不存在 → 错误串。
  - `status:'completed'` → **触发 `TaskCompleted` hook（await 阻塞）**：若 block → 拒绝完成、status 保持原值、返回被拒说明（对齐 CC）。
  - `status:'deleted'` → 软删（§3.4）。
  - `metadata` 合并：拷入给定键，值为 `null` 的键删除。
  - 返回改动字段列表 `已更新任务 #<id>：<改了的字段>`。
- hook payload：`{ hook_event_name:'TaskCompleted', task_kind:'todo', task_id:id, status:'completed', subject }`。

### TaskList
```ts
input: {}
```
- 描述：`列出当前任务清单（id/subject/status）。`
- 行为：`store.list()`（已排除 _deleted 与 metadata._internal）→ 文本：每行 `<mark> #<id> <subject>（<activeForm>）`（mark：completed `✔` / in_progress `▸` / pending `·`）。空清单 → `（任务清单为空）`。`isReadOnly:true`。

## 6. hooks（TaskCreated/TaskCompleted，await 阻塞 + task_kind 全集化）

**must-fix（专家审查）：** 现有后台 dispatch（agent.ts:63/85、bash.ts:58/64/69）是 **fire-and-forget 非阻塞**；todo 侧将用**同名事件 await 阻塞**。若判别符靠「字段缺失」会脆弱且语义混淆（同一 `exit 2` 对后台无效、对 todo 却删除任务）。解法：

1. todo 侧 payload 带 `task_kind:'todo'`，**await** hook 结果并尊重 block（TaskCreated block→删任务+错误；TaskCompleted block→拒绝完成）。
2. 给现有 **5 处后台 dispatch 也补 `task_kind:'background'`**（一行级改动 ×5），令判别符为全集、非靠缺失。后台仍 fire-and-forget 不变。
3. 不引入 `TodoCreated/TodoCompleted` 新事件名（破坏 CC 对齐、hook 作者移植配置失效）。

todo hook 经 `ctx.hookDispatch?.(event, payload)`（①b-1 已建枢纽）。`ctx.hookDispatch` 缺失（不应发生于主会话/headless）→ 视作未 block，正常创建/完成（fail-safe）。

## 12. 记录偏离 / 取舍

1. **每任务一文件**（对齐 CC，用户钦定改自单文件方案）：单 agent 无并发写，并发文件锁/异步队列不做（YAGNI），temp+rename 原子写足够。

3. **metadata + activeForm 都保留**（用户钦定，忠实 CC + 前向兼容）：现阶段 metadata._internal 无生产者、activeForm 仅文本消费者（reminder/TaskList），为将来 TUI/1.6 备。
4. **依赖图 blocks/blockedBy/owner 留 1.6**：多 agent 协作本质，本件单会话不做。
